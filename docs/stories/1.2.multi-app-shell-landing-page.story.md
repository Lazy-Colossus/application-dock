# Story 1.2: Multi-App Shell — Landing Page & Navigation

## Status

Ready for Review

## Story

**As an** operator,
**I want** a mobile-friendly landing page that shows all available apps as large tappable cards,
**so that** I can quickly navigate to any app without reading instructions.

## Acceptance Criteria

1. **Given** the app loads at the root URL, **When** the shell landing page renders, **Then** each entry in `src/apps/registry.ts` appears as an `AppCard` with icon and label; cards are ≥44px tap targets.
2. **Given** a registered app entry in `registry.ts`, **When** an operator taps the app card, **Then** they are routed to the app's route using Vue Router history mode with no full-page reload.
3. **Given** any screen inside any app, **When** the operator looks for a way to return home, **Then** a persistent home control is visible and tapping it returns to the shell landing page (`/`).
4. **Given** a new app is added to `registry.ts`, **When** the shell loads, **Then** the new app card appears automatically — no changes required to `HomePage.vue`, `MainLayout.vue`, or any router file other than adding the lazy-loaded route.
5. **Given** the `GET /api/apps` endpoint, **When** called, **Then** it returns the list of registered apps as JSON (matching `registry.ts` entries); response is a direct array, no envelope wrapper.
6. **Given** a screen width of 360px, **When** the landing page renders, **Then** layout is usable with no horizontal scroll and all tap targets remain ≥44px.

- [x] **Task 7 — `useApi` composable** (foundational; first use here)
  - [ ] Create `frontend/src/composables/useApi.ts` exporting a single HTTP wrapper.
  - [ ] Use `axios` or native `fetch` — single instance with `baseURL: '/api'` (dev proxy or same-origin in prod handles routing)
  - [ ] Expose `get<T>(url)`, `post<T>(url, body)`, `put<T>(url, body)`, `del<T>(url)` returning typed promises
  - [ ] **Error contract (CRITICAL — relied upon by downstream stores):**
    - On non-2xx, throw an `ApiError` (Error subclass) with these fields populated:
      ```ts
      export class ApiError extends Error {
        readonly status: number    // HTTP status code (e.g. 404, 422, 500)
        readonly detail: string    // value of response body's `detail` field, or status text if absent
        readonly body?: unknown    // raw parsed body (if any) for advanced cases
        constructor(status: number, detail: string, body?: unknown) {
          super(`${status}: ${detail}`)
          this.status = status
          this.detail = detail
          this.body = body
        }
      }
      ```
    - Stores branch on `err instanceof ApiError && err.status === 404` (used by Story 3.2 to treat "no in-progress session" as a normal outcome, not an error). Without this contract, that pattern requires inspecting messages — brittle.
    - On network failure (no response — DNS/connection refused), throw an `ApiError(0, 'Network error')` so callers can branch uniformly on `.status`.
  - [ ] Components and stores MUST go through this — no raw `fetch`/`axios` elsewhere [Source: architecture.md#Enforcement Guidelines]

### API contract
- `GET /api/apps` → direct JSON array of `{ id, label, icon, route }` [Source: architecture.md#API & Communication Patterns]
- No envelope: response body is `[ { ... }, { ... } ]`, not `{ "data": [...] }` [Source: architecture.md#Format Patterns]
- Standard FastAPI error shape `{"detail": "..."}` if errors arise (none expected for v1) [Source: architecture.md#API & Communication Patterns]

### App registry pattern (extensibility — AC 4)
- The shell reads `apps` directly from `src/apps/registry.ts`. No store. [Source: architecture.md#Frontend Architecture, "useShellStore removed from scope — registry.ts is a static array"]
- Adding a new app = append one entry to `registry.ts` + add a lazy-loaded route in `router/index.ts`. Nothing else changes. The home page iterates the array; the AppCard is generic.
- The backend `GET /api/apps` is an additional source of truth (mirrors `registry.ts` content). In v1 this is informational and not consumed by the FE shell directly; the FE uses the static `registry.ts`. The endpoint exists per FR/architecture for future integrations.

[Source: EXPERIENCE.md#Component Patterns, #Interaction Primitives]

- Back navigation: top app bar arrow only on non-root routes; follows Vue Router history.

- Tap to act — no long-press, no swipe.
- Banned: pull-to-refresh, carousels, swipe-to-delete.
- Voice/tone: section label "YOUR APPS", page title "Apps" — terse [Source: EXPERIENCE.md#Voice and Tone]
