## Decisions

1. **Manual frontend scaffold (not `npm create quasar@latest`).** The Quasar CLI is interactive and cannot complete in a non-TTY shell. Architecture.md#Selected Approach explicitly endorses "Custom scaffold" as the chosen path. I will produce the exact tree the Quasar CLI would have generated (Vite + Vue 3 + Composition API + TypeScript + Sass), then run `npm install` to fetch deps. Output is deterministic and identical in shape to a CLI scaffold.

2. **Dependency manifest split.** Story Task 6 adds `pytest` + `httpx` (test-only). Per coding-standards.md and to keep the prod image lean, these go in `backend/requirements-dev.txt`, not `backend/requirements.txt`. AC 4 says "`requirements.txt` and `package.json` are the only dependency manifests" — interpreting `requirements.txt` as the canonical *runtime* manifest; `requirements-dev.txt` is a dev-only sibling that the production Dockerfile does NOT install. Source-tree.md lists both files explicitly, so this is consistent with the wider plan.

3. **Docker build context = repo root, dockerfile = `backend/Dockerfile`.** The story called out this ambiguity ("adjust if needed"). Repo-root context is the only choice that allows the multi-stage build to copy both `frontend/` and `backend/` into stages. `docker-compose.yml` will declare `build.context: .` and `build.dockerfile: backend/Dockerfile`.

4. **No Docker verification step.** I will not run `docker compose up`. AC 1 will be marked complete only after user verifies on a host with Docker installed. The Dockerfile + compose file will be written exactly per spec so they're ready when Docker is available.

5. **Python version pin.** Story uses Python 3.12. Dockerfile stage 2 = `python:3.12-slim`. Local dev uses miniconda's 3.12.9 — close enough; pyproject/requirements pins library versions, not interpreter.

6. **Static-files path inside container.** Multi-stage Dockerfile copies `frontend/dist/spa/` → `/app/dist/spa/`. `app/main.py` mounts `StaticFiles(directory="dist/spa")` *relative to working dir* (`/app`). For local dev (no Docker), if `dist/spa/` doesn't exist, the StaticFiles mount would fail at startup — I will guard with an existence check and skip the mount in that case, so `uvicorn app.main:app --reload` works in development before any frontend build.

7. **Carbon theme tokens.** Out of scope for Story 1.1 (they belong to Story 1.2 where the shell UI is built). I'll create empty/minimal `src/css/app.sass` and `src/css/quasar.variables.sass` placeholders so Quasar boots; 1.2 will populate them.

## Risk register

| Risk | Mitigation |
|------|------------|
| `npm install` fails (network, peer-dep conflicts) | Retry with `--legacy-peer-deps` if needed; report failure if unresolvable. |
| Quasar v2 deps changed since pinning | Don't pin versions in `package.json` — use `latest` for `quasar`, `@quasar/extras`, `@quasar/app-vite`. CI will catch breakage when added later. |
| `StaticFiles` mount crashes uvicorn in dev | Guard with `dist/spa/` existence check (decision 6 above). |
| pydantic-settings v2 API differs from v1 | Use v2 syntax (`BaseSettings` from `pydantic_settings`, `SettingsConfigDict`). |
| Path resolution for `core/config.py` `data_dir.mkdir(...)` fails on `/data` (no write perms locally) | In dev, `DATA_DIR` should default to a writable path. Story spec says default `/data`. Compromise: keep `/data` default per spec; let the dev override via `.env` if running outside container. Document this in `.env.example`. |
