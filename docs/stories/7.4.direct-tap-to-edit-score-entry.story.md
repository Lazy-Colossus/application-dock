# Story 7.4: Direct Tap-to-Edit in Score-Entry Drawer

## Status

Ready for Review

## Story

**As an** operator,
**I want** to tap any archer's name or either shot slot to edit it directly,
**so that** correcting a single value is fast and the "back to name" arrow is unnecessary.

## Acceptance Criteria

1. **Given** the drawer's archer chips/list, **When** the operator taps an archer name, **Then** that archer becomes the active entry (jumps directly, forward or backward).
2. **Given** the active archer's two shot slots, **When** the operator taps shot-1 or shot-2, **Then** that slot becomes the active target for the next shot-button press.
3. **Given** a shot slot is selected and a shot-value button is pressed, **When** the press registers, **Then** the value is written to the selected slot (not auto-advanced from slot 0).
4. **Given** the new tap-to-edit model, **When** the drawer renders, **Then** the "← Back to {name}" link is removed.
5. **Given** auto-advance is still desirable for fresh entry, **When** the operator fills shot-1 then shot-2 without manually selecting slots, **Then** entry still advances naturally (shot-1 → shot-2 → next archer) — manual tap selection overrides the cursor.

## Tasks / Subtasks

- [x] **Task 1 — Tappable archer chips** (AC: 1)
  - [x] In `ScoreEntryPanel.vue`, make each archer chip a button that sets `archerIndex` to that archer's index and resets `shotIndex` to `0` (or to the first empty slot).
  - [x] Keep the active-chip highlight (`sep__archer-chip--active`).

- [x] **Task 2 — Tappable shot slots** (AC: 2, 3)
  - [x] Make the two `sep__slot` elements buttons; tapping slot 0 sets `shotIndex = 0`, slot 1 sets `shotIndex = 1`.
  - [x] In `onShotTap(value)`, write the value to `entries[currentArcher][shotIndex]` directly (respect the manually selected slot) instead of the slot-0-then-slot-1 state machine.
  - [x] Add an `--active` style to the selected slot so the operator sees where the next press lands.

- [x] **Task 3 — Preserve natural auto-advance** (AC: 5)
  - [x] After writing shot-1 via a fresh (non-manual) flow, advance `shotIndex` to 1; after shot-2, advance to the next archer with `shotIndex = 0` — but only when the operator hasn't manually re-selected a slot/archer.

Manual taps just reposition `archerIndex`/`shotIndex` before the next press. This satisfies both AC 3 and AC 5.

## Dev Notes

### Previous Story Insights
- **Builds on Stories 7.2/7.3** (drawer lifecycle, confirmed flag, zero-fill). Independent of backend changes beyond what 7.1 provides.
- Story 2.4 implemented the cursor as a strict forward state machine (`archerIndex`, `shotIndex`) with a "← Back to {name}" link and `onBack()`. This story replaces directional back-navigation with **direct addressing**: tap any archer, tap any slot.

### Source references
- `frontend/src/apps/archery/components/ScoreEntryPanel.vue`:
  - Archer chips: the `v-for` over `roster` rendering `sep__archer-chip`.
  - Shot slots: the two `sep__slot` divs (`currentEntry[0]`/`[1]`).
  - State: `archerIndex`, `shotIndex`, `onShotTap`, `onBack`, `canGoBack`, `backTargetName`.
- `frontend/src/apps/archery/components/ShotButton.vue` — emits `tap` with the value.

### Interaction model
- `archerIndex` + `shotIndex` together address exactly one slot. Tapping a chip sets `archerIndex`; tapping a slot sets `shotIndex`. A shot-button press writes to that addressed slot, then auto-advances (slot0→slot1→next archer slot0) for fast sequential entry. Re-tapping is the correction path — no directional back button needed.

### Constraints / anti-patterns
- ❌ Don't keep the back link or `onBack`.
- ❌ Don't lose the active-slot visual affordance — the operator must see where the next press lands.
- ❌ Keep entries panel-local until save (unchanged from Story 2.4 / 7.2).

### Testing
- **Frontend:** Vitest + `@vue/test-utils`. Drive taps via `wrapper.find(...).trigger('click')`; assert `entries` state and active indices.
