# Story 2.1: Session Data Model & Repository

## Status

Draft

## Story

**As a** developer,
**I want** a validated session data model and file repository,
**so that** all session data is read and written consistently and safely throughout the application.

## Acceptance Criteria

1. **Given** the Pydantic schema in `backend/app/schemas/session.py`, **When** a session object is created, **Then** it contains: `label` (str), `created` (ISO 8601 str), `status` (`"in_progress" | "finalised"`), `archers` (ordered list of str), `targets` (list of target objects with `number` and `scores` keyed by archer name, two integers each).
2. **Given** `session_repo.py`, **When** `write_session(path, session)` is called, **Then** it writes to a `.tmp` file first, then calls `os.replace()` to atomically replace the target path; no partial writes are ever visible.
3. **Given** `session_repo.py`, **When** `list_sessions()` is called, **Then** it returns all finalised `.json` files in `DATA_DIR`, sorted newest first by filename, excluding `_in_progress.json`.
4. **Given** `DATA_DIR` contains session files, **When** `read_session(label)` is called, **Then** it returns the parsed `SessionData` object for the matching file, or raises a 404-equivalent if not found.
5. **Given** the `core/config.py` settings, **When** `DATA_DIR` env var is not set, **Then** the default value `/data` is used; the directory is created if it does not exist on startup.

### Data model тАФ canonical session JSON shape
[Source: architecture.md#Data Architecture]

```json
{
  "label": "2026-05-28",
  "created": "2026-05-28T14:00:00Z",
  "status": "finalised",
  "archers": ["Alice", "Bob"],
  "targets": [
    {
      "number": 1,
      "scores": {
        "Alice": [10, 8],
        "Bob": [5, 0]
      }
    }
  ]
}
```

Notes:
- `archers` order = roster order (preserved everywhere)
- `scores` is keyed by archer name (names are unique per session тАФ FR-2.2)
- `targets` contains only **confirmed** targets тАФ absent target numbers are not yet confirmed
- Valid shot values: `{0, 5, 8, 10, 11}` тАФ integers, never strings [Source: architecture.md#Format Patterns]
- Target numbers: 1..18 inclusive [Source: FR-2.5]

### File naming
[Source: architecture.md#Data Architecture]

- Finalised: `YYYY-MM-DD.json`, `YYYY-MM-DD-2.json`, `YYYY-MM-DD-3.json` for repeats on same day
- In-progress: single fixed name `_in_progress.json` (FR-3.8: only one at a time)
- All files live in `DATA_DIR` (env, default `/data`)

### Atomic write strategy
[Source: architecture.md#Data Architecture, #Enforcement Guidelines]

- Write to `path.with_suffix(".tmp")` first, then `os.replace(tmp, path)` to commit atomically
- `os.replace()` is atomic on POSIX; best-effort on Windows (acceptable for v1 LAN use)
- This applies to BOTH the in-progress temp file and the finalised session files
- The previously valid `_in_progress.json` remains intact if the server is interrupted mid-write (Story 3.1 AC depends on this guarantee)

### Layering rules (CRITICAL)
[Source: architecture.md#Service Boundaries, #Enforcement Guidelines]

- **Repositories: file I/O ONLY.** No business logic, no HTTP concerns.
- **Services: business logic ONLY.** Compute scores, generate labels, decide which file to write тАФ but call repositories for actual file ops.
- **Routers: HTTP only.** Parse request тЖТ call service тЖТ return response. No file I/O, no logic.
- `session_repo.py` is the ONLY module that touches the filesystem [Source: architecture.md#Data Boundaries]
- No HTTP exceptions in the repo тАФ raise standard Python exceptions (`FileNotFoundError`, `ValueError`); the router translates to HTTP status codes.

### Config (re-confirms Story 1.1 work)
[Source: architecture.md#Data Architecture, epics.md FR-3.4]

- `app/core/config.py` uses `pydantic-settings`:
  ```python
  from pathlib import Path
  from pydantic_settings import BaseSettings, SettingsConfigDict

  class Settings(BaseSettings):
      model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8")
      data_dir: Path = Path("/data")

  settings = Settings()
  settings.data_dir.mkdir(parents=True, exist_ok=True)
  ```
- Env var name: `DATA_DIR` (uppercase). Pydantic-settings auto-uppercases attribute names when reading env.
- `DATA_DIR` is the only required setting in v1.
