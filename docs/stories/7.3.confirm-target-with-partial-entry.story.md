# Story 7.3: Confirm Target With Partial Entry

## Status

Draft

## Story

**As an** operator,
**I want** to confirm a target even if I haven't entered every shot,
**so that** I can quickly mark a target done and have missing shots count as 0.

## Acceptance Criteria

1. **Given** the score-entry drawer, **When** it renders, **Then** **Confirm Target** is always enabled (no longer gated on all shots being filled).
2. **Given** some shots are empty, **When** the operator taps **Confirm Target**, **Then** every empty shot is saved as `0` and the target is marked `confirmed: true`.
3. **Given** a target is confirmed, **When** the board re-renders, **Then** that target icon is green.
4. **Given** a confirmed target is re-opened, **When** the drawer renders, **Then** the previously saved values (including any auto-filled 0s) are shown and editable; re-confirming re-saves.
5. **Given** the confirm action, **When** it persists, **Then** the saved target passes in-progress validation (Story 7.1) — partial 0-filled values are valid in-progress.

- [ ] **Task 3 — Green reflects `confirmed`** (AC: 3)
  - [ ] Ensure `store.isConfirmed(n)` returns the target's `confirmed` flag (set up in Story 7.2). A target saved on X-close (`confirmed:false`) is NOT green; a confirmed one is.

- [ ] **Task 4 — Re-open shows saved values** (AC: 4)
  - [ ] `initEntries()` already pre-populates from `store.targetByNumber(n)`. Confirm it reads the saved `scores` (now possibly containing auto-filled `0`s) and that re-confirm re-saves.

- [ ] **Task 5 — Tests** (AC: 1–5)
  - [ ] `ScoreEntryPanel.spec.ts`: Confirm enabled with zero entries; confirming with partial entry sends full-roster scores with `0` for blanks and `confirmed:true`; re-open shows saved values; confirm disabled only while `store.loading`.

## Dev Notes

### Previous Story Insights
- **Depends on Story 7.1** (in-progress validation accepts 0-filled full-roster targets; `confirmed` field) and **Story 7.2** (drawer lifecycle, `confirmed` plumbing, save-on-close).
- Story 2.4 gated Confirm on `allComplete` (every archer has both shots). This story removes that gate and introduces zero-fill at confirm time.
- `onConfirm()` currently only includes archers with both shots filled in `scores`. Change it to **always** include every archer with `0` for blanks — otherwise in-progress validation (2 shots per listed archer) and the green/finalise logic break.

### Source references
- `frontend/src/apps/archery/components/ScoreEntryPanel.vue` — `onConfirm`, `allComplete`, `entries`, confirm button `:disable`.
- `frontend/src/apps/archery/stores/useArcherySessionStore.ts` — `saveTarget`, `isConfirmed`, `targetByNumber`.
- Backend (Story 7.1): in-progress targets accept `0`/`null`; finalise also zero-fills (defence in depth).

### Why zero-fill at confirm AND at finalise
Confirm-time zero-fill makes the board state honest (a confirmed target is fully scored and green). Finalise-time zero-fill (Story 7.1) is the safety net for targets the operator confirmed-as-partial or never opened. Both must agree: unentered → `0`.

### Constraints / anti-patterns
- ❌ Don't leave a confirmed target with `null` shots — confirm materialises blanks to `0`.
- ❌ Confirm sets `confirmed: true`; X-close sets `confirmed: false` (Story 7.2).
- ❌ All HTTP via `useApi.ts` through the store.

### Testing
- **Frontend:** Vitest. Assert `saveTarget` payload on confirm includes all roster archers with `0`-filled blanks and `confirmed:true`.
