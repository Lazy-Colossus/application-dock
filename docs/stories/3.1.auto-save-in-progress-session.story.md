# Story 3.1: Auto-Save In-Progress Session

## Status

Draft

## Story

**As an** operator,
**I want** the session to be automatically saved to the server after every confirmed target,
**so that** no scoring data is lost if the browser closes or the device loses connectivity.

## Acceptance Criteria

1. **Given** the operator confirms a target on the scoring board, **When** the confirmation is processed, **Then** `PUT /api/archery/sessions/in-progress` is called with the full current session state; the store's `saveTarget()` action handles this automatically — no manual save step required.
2. **Given** the `PUT /api/archery/sessions/in-progress` endpoint, **When** it receives a valid session payload, **Then** `session_repo.write_in_progress(session)` is called, which writes to `_in_progress.tmp` then atomically replaces `_in_progress.json` via `os.replace()`.
3. **Given** the in-progress file is being written, **When** the server process is interrupted mid-write, **Then** the previously valid `_in_progress.json` remains intact (atomic write guarantee).
4. **Given** only one in-progress session is allowed, **When** `_in_progress.json` already exists and a new target is confirmed, **Then** the file is overwritten (not appended); the single-file constraint is always maintained.
5. **Given** the `useArcherySessionStore`, **When** `saveTarget()` is called, **Then** it sets `loading = true` before the request and `loading = false` in the `finally` block; any error lands in `store.error`.

### API contract — `PUT /api/archery/sessions/in-progress`
[Source: architecture.md#API & Communication Patterns]

- **Request body:** Full `SessionData` JSON — `{ label, created, status, archers, targets }`
- **Success (200 OK):** Echo of the saved `SessionData` (server may normalize `status` to `"in_progress"`)
- **422 Unprocessable Entity:** Validation failure (e.g. invalid shot value, duplicate archers, empty archer list)
- **No envelope.** Direct serialization.

### Atomic write — guarantees & failure modes
[Source: architecture.md#Data Architecture, #Enforcement Guidelines]

- `_atomic_write_json(path, payload)` (Story 2.1):
  1. Writes JSON to `path.with_suffix(".tmp")` (i.e. `_in_progress.json.tmp`)
  2. Calls `os.replace(tmp, path)` — atomic on POSIX
- On POSIX, if the process is killed between step 1 and step 2, only the `.tmp` artifact exists — `_in_progress.json` is unchanged
- On Windows, `os.replace()` is best-effort atomic — sufficient for v1 LAN use
- Concurrency note: only one in-progress session at a time (FR-3.8). The single fixed filename `_in_progress.json` is overwritten on every PUT.
- Single-writer assumption: there is no multi-device write coordination in v1. If two LAN devices both PUT simultaneously, last-write-wins. The PRD explicitly accepts this as out of scope (no real-time multi-device editing). [Source: architecture.md#Areas for Future Enhancement — "Real-time concurrent scoring (WebSocket) for multi-device entry"]

### Save-error UX
[Source: EXPERIENCE.md#State Patterns]

| State | Treatment |
|-------|-----------|
| Saving | Confirm button shows QSpinner. No toast. Silent on success. |
| Save error | QBanner top-of-page: "Couldn't save. Check connection." Retry button calls `store.saveTarget(lastTarget)` |

The board's local mutation (target appearing green) is OPTIMISTIC in the current flow — after `upsertTarget` runs, `store.session.targets` already contains the new entry BEFORE the network resolves. If the save fails:
- The current implementation in Task 3 OVERWRITES `session.value = saved` on success only — on failure the local optimistic mutation persists, but `store.error` is set.
- Recommendation: on failure, KEEP the optimistic mutation (so the user's last action is preserved visually) and surface the banner. The retry button resubmits.
- Acceptable alternative: roll back the local mutation on failure (`session.value = previousSession`) — slightly more code, more "honest" UI. Pick one and document.
  - **This story picks: keep optimistic mutation on failure**, surface QBanner, allow retry. The user gets a clear "save didn't go through" signal without losing visual progress.

### State management pattern (re-confirmation)
[Source: architecture.md#State Management Patterns, #Enforcement Guidelines]

```ts
async function action() {
  loading.value = true
  error.value = null
  try {
    // HTTP call via useApi
  } catch (e) {
    error.value = String(e)
  } finally {
    loading.value = false
  }
}
```
- Every async store action follows this shape — `saveTarget` already does
- `loading` and `error` refs are exposed by the store
- All HTTP through `useApi.ts` — never raw `fetch`/`axios` [Source: architecture.md#Enforcement Guidelines]

### Constraints / anti-patterns
- ❌ Do not call `fetch` in the component — go through the store, which uses `useApi` [Source: architecture.md#Enforcement Guidelines]
- ❌ Do not skip `loading`/`error`/`finally` pattern [Source: architecture.md#Enforcement Guidelines]
- ❌ Do not write to `_in_progress.json` from the router — go through the service → repo layering [Source: architecture.md#Service Boundaries]
- ❌ Do not change `_atomic_write_json` to skip the `.tmp` step [Source: architecture.md#Enforcement Guidelines]
- ❌ Do not show a toast on successful save — silent UX (NFR/UX preference) [Source: EXPERIENCE.md#State Patterns]
- ❌ Do not block the user from confirming the next target while a save is in flight — let saves stack/queue OR (simpler) allow concurrent local mutations + serial HTTP calls. v1 acceptable approach: each Confirm Target triggers its own PUT; if the network is slow, the spinner remains but the board state is updated optimistically. Document the trade-off.
