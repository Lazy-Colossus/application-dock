# Story 7.2: Score-Entry Drawer Lifecycle

## Status

Draft

## Story

**As an** operator,
**I want** the score-entry drawer to stay hidden until I tap a target and to save whenever it closes,
**so that** the board stays clean and I never lose an entry.

## Acceptance Criteria

1. **Given** the scoring board, **When** it first renders, **Then** the score-entry drawer is hidden.
2. **Given** the operator taps a target number, **When** the tap registers, **Then** the drawer opens for that target with any previously entered scores pre-populated.
3. **Given** the drawer is open, **When** the operator taps the **X** (top-right), **Then** the drawer closes and the current entry is saved (partial entries persisted as-is; see Story 7.3 for zero-fill on Confirm).
4. **Given** the drawer is open, **When** the operator taps **Confirm Target**, **Then** the drawer closes and the entry is saved (with zero-fill per Story 7.3).
5. **Given** the entry is saved on close, **When** the save completes, **Then** `PUT /api/archery/sessions/in-progress/{label}` has persisted the target; the discard-guard dialog from Story 2.4 is removed (closing always saves, never discards).
6. **Given** a save fails, **When** the error is surfaced, **Then** the board shows the existing error banner and the entry is not lost from local state.

## Dev Notes

### Previous Story Insights
- **Depends on Story 7.1** (nullable shots + `confirmed` flag in schema/types) and **Story 6.1** (`PUT /api/archery/sessions/in-progress/{label}` label-scoped). The store's `saveTarget` must target the active session's label.
- Story 2.4 built `ScoreEntryPanel.vue` with: panel-local `entries`, `archerIndex`/`shotIndex` cursor, a discard-guard dialog (`requestClose`/`forceClose`), and `:no-backdrop-dismiss="hasAnyEntry"`. **This story removes the discard guard** — closing always saves.
- `store.saveTarget(target)` already does optimistic local upsert + `PUT` + `loading`/`error` + rethrow. With Story 6.1 it must `PUT` to `/archery/sessions/in-progress/{session.label}`.

### Source references
- `frontend/src/apps/archery/components/ScoreEntryPanel.vue` — `requestClose`, `forceClose`, `discardGuardOpen`, `onConfirm`, `initEntries`, `hasAnyEntry`.
- `frontend/src/apps/archery/pages/ScoringBoardPage.vue` — error banner, `isConfirmed`, `store.openTarget`.
- `frontend/src/apps/archery/stores/useArcherySessionStore.ts` — `saveTarget`, `openTarget`, `closeTarget`, `isConfirmed`, `targetByNumber`.
- `frontend/src/apps/archery/types.ts` — `TargetScores`.

### Save semantics (close vs confirm)
- **X (close):** persist whatever is entered, `null` for blanks, `confirmed: false`. The target may show as *not* green (it's saved but unconfirmed).
- **Confirm Target:** zero-fill blanks → `0`, `confirmed: true`, target goes green (Story 7.3).
- Both close the drawer. There is no "discard" path anymore.

### Constraints / anti-patterns
- ❌ Don't zero-fill on X-close — that's Confirm-only (preserves the unentered/entered-zero distinction mid-session).
- ❌ Don't reintroduce a discard dialog.
- ❌ All HTTP via `useApi.ts` (through the store).

### Testing
- **Frontend:** Vitest + `@vue/test-utils` + `@pinia/testing`. Stub `saveTarget`; assert call args (`confirmed` flag, null preservation) and that `closeTarget` follows.
