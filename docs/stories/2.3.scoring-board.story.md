# Story 2.3: Scoring Board

## Status

Draft

## Story

**As an** operator,
**I want** a visual board of 18 numbered target icons I can tap in any order,
**so that** I can score targets as my group reaches them on the field.

## Acceptance Criteria

1. **Given** the operator is on the scoring board, **When** it renders, **Then** 18 target icons numbered 1–18 are displayed; each is at minimum 44px in both dimensions.
2. **Given** a target has not yet been confirmed, **When** it is displayed, **Then** the icon appears in its default (unconfirmed) state.
3. **Given** a target has been confirmed, **When** it is displayed, **Then** the icon appears green.
4. **Given** the operator taps any unconfirmed target, **When** the tap registers, **Then** the score-entry panel opens for that target.
5. **Given** the operator taps a confirmed (green) target, **When** the tap registers, **Then** the target re-opens in the score-entry panel with its previously entered scores pre-populated; the green state resets until re-confirmed.
6. **Given** a screen width of 360px, **When** the scoring board renders, **Then** all 18 icons are visible and tappable without horizontal scrolling.

### Visual specification
[Source: DESIGN.md#Components — target-icon, EXPERIENCE.md#State Patterns]

- **Target icon size:** 56px × 56px, `rounded/sm` (8px) [Source: DESIGN.md frontmatter `components.target-icon`]
- **Unconfirmed:** `surface-card` bg, `border` outline, target number in `ink-secondary` using `score` typography (Roboto Mono 700 20px, but mockup uses 18px — go with mockup HTML at 18px since it's the rendered source of truth)
- **Confirmed:** `confirmed` fill (#2E7D32), no outline, number in `confirmed-text` (#A5D6A7)
- **Grid:** 3 columns × 6 rows on mobile, `gap: 8px`, page padding 16px [Source: scoring-board.html mockup]
- **Subtitle bar:** centered, `meta` typography (12px in DESIGN.md, 13px in mockup — match mockup), `ink-secondary` color, `surface-base` bg, 8px vertical padding
- **Archer chips strip:** below the grid, `surface-card` bg, `rounded/sm`, 32px height in mockup (`meta` typography); chips do NOT show a remove icon here (roster is locked per FR-2.3)
- **Desktop (≥768px):** Grid expands to 6 columns × 3 rows [Source: EXPERIENCE.md#Responsive & Platform]

### Behavioral spec
[Source: EXPERIENCE.md#Component Patterns, #State Patterns]

- **Tap unconfirmed target:** opens Score Entry Panel for that target
- **Tap confirmed (green) target:** opens Score Entry Panel with previously entered scores pre-populated; the green state resets until the operator re-confirms (FR-2.12; visual reset is handled by the panel's confirm flow in Story 2.4)
- **Targets can be entered in any order** (FR-2.6) — no enforced sequence
- **All 18 confirmed → "View Results" banner:** NOT in this story — implemented in Story 2.5 along with the results screen
- **No long-press, no swipe** — tap only [Source: EXPERIENCE.md#Interaction Primitives]
- **Reduce motion:** if user has `prefers-reduced-motion`, the panel should appear without slide-up animation (Story 2.4's responsibility — flag it here but implement in 2.4)

### Accessibility floor
[Source: EXPERIENCE.md#Accessibility Floor]

- Each target icon: `<button>` element, `aria-label="Target {n}, {confirmed|not confirmed}"`
- Focus traversal: 1 → 2 → 3 → ... → 18 in reading order
- Confirmed state is communicated by both color AND number label (number stays visible); no color-only signal
- Tap targets ≥44px (56px in this design)

### Data flow
[Source: architecture.md#Data Flow — Score entry → persistence]

Score-entry persistence and the "save target → update green" round-trip is implemented in **Story 2.4** (component) and **Story 3.1** (HTTP call via `saveTarget()`). This story does NOT make any HTTP requests itself — it only reads the `session` ref in the store and reacts to user taps by opening the panel.

### Constraints / anti-patterns
- ❌ Do not call `fetch`/`axios` here — no HTTP from this story (saves happen in 2.4 + 3.1) [Source: architecture.md#Enforcement Guidelines]
- ❌ Do not place `TargetIcon.vue` under `src/components/` — it's archery-specific [Source: architecture.md#Anti-Patterns to Avoid]
- ❌ Do not add a Pinia store for board state — reuse `useArcherySessionStore` [Source: architecture.md#State Management Patterns]
- ❌ Do not use box shadows for confirmed/active states — color steps only [Source: DESIGN.md#Elevation & Depth]
- ❌ Do not pre-render unconfirmed `TargetScores` entries in `session.targets` — absence == unconfirmed [Source: architecture.md#Data Architecture]

### Testing

- **Frontend tests:** Vitest + Vue Test Utils, co-located `*.spec.ts` files
- **Manual verification:**
  - From Session Setup with `["Alice", "Bob", "Jamie"]` → arrive on Scoring Board
  - 18 targets visible, all gray; subtitle reads "0 of 18 confirmed"
  - Archer chips read "Alice", "Ben", "Jamie" below the grid (no X icons)
  - Tap target 7 → stub panel opens; close panel
  - At 360px viewport: no horizontal scroll; all 18 targets in 3-col × 6-row grid; chips wrap if needed
  - Refresh (or hot-reload) without a session in `store.session` → page redirects to `/archery`
- **Edge cases:**
  - Long archer names (e.g. 30 chars): chips should wrap; not break grid layout
  - High-DPI displays: target numbers render crisply in Roboto Mono
  - Re-mount: state in the Pinia store persists across navigations within the SPA
