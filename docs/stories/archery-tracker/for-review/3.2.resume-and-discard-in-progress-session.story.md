# Story 3.2: Resume & Discard In-Progress Session

## Status

Ready for Review

## Story

**As an** operator,
**I want** to be offered the choice to resume or discard an in-progress session when I open the app,
**so that** I can recover from an accidental browser close without losing scored targets.

## Acceptance Criteria

1. **Given** `_in_progress.json` exists in `DATA_DIR`, **When** the operator opens the Archery home screen, **Then** a prompt is displayed offering "Resume Session" and "Start Fresh (Discard)" — it does not auto-navigate; the operator must choose.
2. **Given** the operator taps "Resume Session", **When** the action completes, **Then** `GET /api/archery/sessions/in-progress` returns the full session state; the scoring board renders with all previously confirmed targets green and all unconfirmed targets open — exact state restored.
3. **Given** a device on the same LAN (not the original device), **When** it opens the Archery home screen while `_in_progress.json` exists, **Then** the same resume prompt appears and resuming restores the full board state — state is server-side, not browser-local.
4. **Given** the operator taps "Start Fresh (Discard)", **When** the action completes, **Then** `DELETE /api/archery/sessions/in-progress` is called; `_in_progress.json` is deleted; the operator proceeds to session setup as if starting fresh.
5. **Given** no `_in_progress.json` exists, **When** the operator opens the Archery home screen, **Then** no resume prompt appears; the normal home screen is shown with "New Session" and "History".
6. **Given** `GET /api/archery/sessions/in-progress`, **When** no in-progress file exists, **Then** the endpoint returns 404; the frontend treats this as "no session to resume".

    ```ts

    async function checkInProgress(): Promise<SessionData | null> {
      loading.value = true
      error.value = null
      try {
        const s = await api.get<SessionData>('/archery/sessions/in-progress')
        return s
      } catch (e) {
        if (e instanceof ApiError && e.status === 404) return null  // expected when nothing to resume

        throw e
      } finally {
        loading.value = false
      }
    }
    ```

### API contracts
[Source: architecture.md#API & Communication Patterns]

**`GET /api/archery/sessions/in-progress`**
- 200 OK: `SessionData` body (status `"in_progress"`)
- 404 Not Found: `{"detail": "No session in progress."}`
- No envelope.

**`DELETE /api/archery/sessions/in-progress`**
- 204 No Content (file deleted OR was already absent — idempotent)
- No body, no envelope.

### Session hydration on resume — exact state restoration
[Source: epics.md FR-3.7, architecture.md#Data Flow — Session resume]

```
ArcheryHomePage mounts
  → useArcherySessionStore.checkInProgress()
    → GET /api/archery/sessions/in-progress
      → session_repo.read_in_progress() → SessionData
        → store.session = SessionData
          → ScoringBoardPage iterates store.session.targets
            → green targets re-rendered from non-null entries
```

Because the session JSON contains the FULL roster + every confirmed target with its scores (Story 2.1 schema), the scoring board re-renders identically — confirmed targets green, unconfirmed targets open. No additional API calls required to rebuild state.

### LAN multi-device behavior (AC 3)
[Source: architecture.md#Decision Impact Analysis, EXPERIENCE.md#Key Flows — Flow 2]

- State is server-side; ANY device that reaches the LAN host can `GET /api/archery/sessions/in-progress` and resume
