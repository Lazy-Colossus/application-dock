# Story 6.3: Archery Home Redesign — New Session / Resume / History

## Status

Ready for Review

## Story

**As an** operator,
**I want** New Session, Resume, and History as standard buttons on the Archery home screen,
**so that** I can choose my action directly instead of being forced into a single resume/discard prompt.

## Acceptance Criteria

1. **Given** the Archery home screen, **When** it renders, **Then** it shows three primary buttons — **New Session**, **Resume**, **History** — each a ≥44px tap target; the previous persistent resume/discard bottom sheet is removed.
2. **Given** the operator taps **New Session** and no session is in progress, **When** the tap registers, **Then** they go straight to session setup (equivalent to the old "Start Fresh").
3. **Given** at least one session is in progress, **When** the operator taps **New Session**, **Then** a popup appears asking whether to **Delete in-progress session(s)** or **Leave them**; choosing Delete removes all in-progress sessions then proceeds to setup; choosing Leave proceeds to setup leaving them intact.
4. **Given** the operator taps **History**, **When** the tap registers, **Then** they navigate to the history page (unchanged behaviour).
5. **Given** the home screen, **When** it loads, **Then** it queries in-progress sessions once and reflects their presence in the Resume button's visibility (see Story 6.4) without auto-opening any sheet.

## Dev Notes

### Previous Story Insights
- **Depends on Story 6.1** (`GET /api/archery/sessions/in-progress` returns an array; per-label `DELETE`). **Coordinates with Story 6.4** (Resume button visibility + picker) — 6.3 lays out the buttons and the New-Session flow; 6.4 wires Resume's gating and picker. Land 6.3 then 6.4, or together.
- The current `ArcheryHomePage.vue` shows New Session + History buttons AND a **persistent** resume bottom sheet that auto-opens on mount when `_in_progress.json` exists (Story 3.2). This story removes that sheet entirely; Resume becomes an explicit button.

### Source references
- `frontend/src/apps/archery/pages/ArcheryHomePage.vue` — current buttons + resume sheet + `onMounted` `checkInProgress`.
- `frontend/src/apps/archery/stores/useArcherySessionStore.ts` — `checkInProgress`, `resumeSession`, `discardSession` (single-session); add list-based actions here.
- `frontend/src/apps/archery/pages/ResultsPage.vue` — reference for `q-dialog` styling.
- Architecture: `docs/architecture/source-tree.md` — stores expose `loading`/`error`; all HTTP via `useApi.ts`.

### "Delete it or leave it be" interpretation
Per FR-6.4 and the confirmed multi-session model: New Session always proceeds to setup. The popup only appears when ≥1 session is open and offers **Delete in-progress session(s)** (removes all open sessions) vs **Leave them** (both/all coexist). This is intentional friction to discourage stale-session buildup while still allowing concurrency.

### Constraints / anti-patterns
- ❌ No auto-opening sheet/dialog on load — the screen must be three calm buttons.
- ❌ All HTTP through `useApi.ts`; no raw fetch.
- ❌ Stores never call other stores; the component orchestrates navigation.

### Testing
- **Frontend:** Vitest + `@vue/test-utils` + `@pinia/testing`. Mock the store; assert routing via a mocked `useRouter().push`. Reuse the patterns already in `ArcheryHomePage.spec.ts`.
