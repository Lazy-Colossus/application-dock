# Story 2.2: Session Setup — Roster Entry

## Status

Ready for Review

## Story

**As an** operator,
**I want** to start a new session and enter the archer roster before scoring begins,
**so that** scores are correctly attributed to each archer throughout the session.

## Acceptance Criteria

1. **Given** the operator is on the Archery app home screen, **When** they tap "New Session", **Then** they are taken to the session setup screen with an empty roster and a name input field.
2. **Given** the session setup screen, **When** the operator enters a non-empty, unique name and confirms, **Then** the name is added to the roster list; minimum 1 archer is required to proceed.
3. **Given** the operator enters a name that already exists in the roster, **When** they attempt to add it, **Then** an error message is shown and the duplicate is not added.
4. **Given** the operator enters an empty name, **When** they attempt to add it, **Then** the entry is rejected with a validation message.
5. **Given** at least one archer is in the roster, **When** the operator confirms the roster, **Then** they are taken to the scoring board and the roster is locked — no further additions or removals are possible.
6. **Given** a new session is created, **When** the `POST /api/archery/sessions` endpoint is called, **Then** it returns the session object including an auto-generated label (`YYYY-MM-DD` for the first session of the day, `YYYY-MM-DD #2` etc. for subsequent); the label logic lives in `archery_service.py`.

### API contract — `POST /api/archery/sessions`
[Source: architecture.md#API & Communication Patterns]

- **Request body:** `{ "archers": ["Alice", "Bob"] }` — JSON, snake_case fields
- **Success (200 OK):** Full `SessionData` object (status `"in_progress"`, label auto-generated, targets `[]`, archers as posted)
- **409 Conflict:** When an in-progress session already exists. Body: `{"detail": "A session is already in progress."}`
- **422 Unprocessable Entity:** Validation failure (empty list, duplicate names, empty name strings) — FastAPI default error shape
- No envelope wrapping. Direct serialization. [Source: architecture.md#Format Patterns]

### Label generation rules (FR-2.4)
[Source: epics.md FR-2.4, architecture.md#Data Architecture]

- First session of the day: `YYYY-MM-DD` (e.g. `2026-05-28`)
- Subsequent sessions: `YYYY-MM-DD-2`, `YYYY-MM-DD-3`, … (file naming convention)
- The PRD/epic text uses `#2`, `#3` for **display** purposes; the **filename and stored label** use `-2`, `-3`. Display formatting (replacing `-N` with `#N`) is a frontend display concern — out of scope for this story; can be handled by a `useSessionLabel.ts` composable later.
- For this story: the backend returns the stored label (`-2` form). Frontend displays it as-is for v1; display polish deferred.
- "Day" is determined by server clock (UTC). If a session starts at 23:55 UTC, midnight changes the day mid-session — this is acceptable; the label is set once at creation.

### Behavioral spec (EXPERIENCE.md)

- **Voice:** terse — "New Session", "Add", "Confirm Roster", "Archer name is already used.", "Archer name is required." (one sentence, period, no blame, no emojis)
- **Roster empty state:** "Confirm Roster" button disabled
- **Duplicate name:** Inline error "Archer name is already used."; input border `error` color; Confirm Roster button remains enabled if other archers exist (the error blocks only the duplicate add, not the overall flow)
- **No long-press, no swipe** — to remove an archer pre-confirmation, tap the X icon explicitly
- **Once roster confirmed:** roster is locked. The X icons disappear from chips on subsequent screens (Story 2.3 onward), and there is no UI to add/remove archers (FR-2.3)

### State management pattern
[Source: architecture.md#State Management Patterns]

- `useArcherySessionStore` MUST expose `loading`, `error`, `session` refs
- Every async action sets `loading = true` before, `loading = false` in `finally`
- All HTTP via `useApi.ts` — never raw `fetch`/`axios` [Source: architecture.md#Enforcement Guidelines]
- Component handles `store.error` display (QBanner at top of page) — `console.error` alone is never the only handling [Source: architecture.md#Error Handling]
