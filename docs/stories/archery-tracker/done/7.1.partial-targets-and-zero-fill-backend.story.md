# Story 7.1: Backend — Partial Targets, Confirmed Flag & Zero-Fill

## Status

Ready for Review

## Story

**As a** developer,
**I want** in-progress targets to hold partial scores and a confirmed flag, and finalise/results to treat anything unentered as 0,
**so that** operators can save mid-entry, confirm partial targets, and view results at any time.

## Acceptance Criteria

1. **Given** the `TargetScores` schema, **When** an in-progress target is validated, **Then** shot values may be `null` (length-2 list of `int | null`) and a `confirmed: bool` field (default `false`) is present.
2. **Given** an in-progress session, **When** validated, **Then** targets need NOT cover the full roster and shots need NOT be filled — the strict "keys must equal roster / exactly 2 valid shots" rule applies only to `finalised` sessions.
3. **Given** a finalised session is produced (`finalise_in_progress`), **When** it is written, **Then** every target 1–18 is present, every archer has two shots, and all `null`/missing values are coerced to `0`; `confirmed` is irrelevant once finalised.
4. **Given** `archery_service` results/summary computation, **When** totals are computed for an in-progress or finalised session, **Then** missing targets, missing archers, and `null` shots all contribute `0`.
5. **Given** finalised JSON on disk, **When** read back, **Then** it still validates under the strict finalised rules (no nulls, full roster, 18 targets) — back-compatible with Stories 2.1/2.5.
6. **Given** the `PUT /api/archery/sessions/in-progress/{label}` endpoint, **When** it receives a session with partial/`null` shots and `confirmed` flags, **Then** it persists them without error.

## Dev Notes

### Previous Story Insights
- **Land after Story 6.1** — both edit `schemas/session.py` and `archery_service.py`. 6.1 changes in-progress *storage* (per-label files) and `finalise_in_progress(label)` signature; this story changes target *content* rules and zero-fill.
- Current strict rules (Story 2.1): `TargetScores._validate_scores` requires exactly 2 shots, each in `{0,5,8,10,11}`; `SessionData._validate_target_archers` requires every target's `scores` keys to equal the roster. These are correct for **finalised** sessions but block partial in-progress saves — hence the status-aware split.
- Frontend `TargetScores.scores` is typed `Record<string, [number, number]>`. Update `types.ts` to `[number | null, number | null]` and add `confirmed?: boolean` (done in Stories 7.2/7.3 FE).

### Source references

- `backend/app/services/archery_service.py` — `finalise_in_progress`, `_archer_total`, `_summarise`, `list_history`.
- `backend/app/routers/archery.py` — `PUT`/`finalise` endpoints.
- Architecture: `docs/architecture/source-tree.md` (layering), `tech-stack.md` (Pydantic v2).

### The confirmed vs partial distinction
- `confirmed: bool` per target is the source of truth for the green board state (replaces the old "target exists in `targets[]` == confirmed" heuristic). A target may exist with partial scores yet `confirmed: false` (saved-on-close but not confirmed) — see Stories 7.2/7.3.
- On finalise, everything not entered becomes `0` and the session is fully materialised, so `confirmed` no longer matters for finalised reads.

### Constraints / anti-patterns
- ❌ Don't loosen finalised validation — finalised files must stay strict (Story 2.5 contract; history readers depend on it).
- ❌ Keep all file I/O in `session_repo.py`; zero-fill logic lives in the service.
- ❌ Don't store `0` to mean "unentered" in-progress — use `null`. `0` is a legitimate shot value; conflating them loses information mid-session.

### Testing
- **Backend:** pytest. Cover schema (param both statuses), service zero-fill/materialisation, and finalised round-trip validity. Use temp `DATA_DIR`.

### Completion Notes List

- **Schema (Task 1):** `TargetScores.scores` is now `dict[str, list[int | None]]` and gains `confirmed: bool = False`. `_validate_scores` still requires exactly 2 entries per archer but allows `None` (non-null values must still be in `{0,5,8,10,11}`). The model validator (`_validate_targets`) is now status-conditional: in-progress is lenient (partial roster + null shots allowed); finalised is strict (every present target must cover the full roster with two non-null shots). Stray archers (not in roster) are rejected in **both** modes.
- **Design choice:** the strict "full roster / no nulls" rule is enforced per *present* target rather than requiring exactly 18 targets at the schema layer. This keeps reads of minimal historical/seeded finalised sessions (e.g. `targets: []`) valid, while the service guarantees 18 materialised targets at finalise time. The "18 targets" guarantee is therefore a finalise-service concern, not a read-validation gate (avoids breaking `GET /sessions/{label}` and history reads).
- **Service (Task 2):** `finalise_in_progress` now calls a new `_materialise_targets` helper that builds targets 1–18, fills every missing target/archer/shot with `0`, and marks them `confirmed=True`. `_archer_total` is null/missing-safe (`shot or 0`, skips absent archers) so totals work for in-progress sessions too.
- **Router (Task 3):** No change needed — `PUT /sessions/in-progress/{label}` already takes a `SessionData` body, which now accepts null shots + `confirmed` via the schema.
- **Tests (Task 4):** Updated the two strict-roster schema tests to target finalised status and the new messages; replaced the finalise `targets == []` assertion with zero-fill coverage (18 all-zero targets; partial target keeps entered shots, rest → 0). Added schema tests (confirmed default/set, null shots accepted in-progress / rejected finalised, partial roster in-progress) and a null/missing-safe `_archer_total` unit test.
- **Frontend:** none. Story 7.1 is backend-only; the FE `TargetScores` type and null/`confirmed` handling are introduced in Stories 7.2/7.3. The 6.1 FE shim still sends complete targets, which validate fine under the relaxed in-progress rules.
- **Verification:** `pytest` 141 passed; `ruff`/`black` clean; `vitest run` 109 passed (unchanged).
