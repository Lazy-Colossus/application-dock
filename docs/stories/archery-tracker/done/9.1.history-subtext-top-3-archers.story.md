# Story 9.1: History Subtext — Top 3 Archers

## Status

Done

## Story

**As an** operator,
**I want** each history entry to show the top 3 archers and their scores,
**so that** I can recall a round's standings at a glance without opening it.

## Acceptance Criteria

1. **Given** the `SessionSummary` schema, **When** a summary is produced, **Then** it includes a `top_archers` field: an ordered list (max 3) of `{ name, score }`, highest first, using the existing tie-break (score desc, then case-insensitive name).
2. **Given** `archery_service._summarise`, **When** it computes a summary, **Then** `top_archers` holds up to the first 3 ranked archers; sessions with fewer than 3 archers list only those present.
3. **Given** `GET /api/archery/sessions`, **When** called, **Then** each row includes `top_archers`; existing `winner`/`winning_score` remain for back-compatibility (winner == top_archers[0]).
4. **Given** `HistoryListItem.vue`, **When** a row renders, **Then** the subtext shows the archer count followed by the top 3 as `Name Score` separated by ` · ` (e.g. `5 archers · Alice 372 · Bob 350 · Cara 318`) instead of only the winner; the `aria-label` is updated accordingly.
5. **Given** the frontend `SessionSummary` type, **When** updated, **Then** it includes `top_archers`; existing specs are updated to the new subtext.

## Dev Notes

### Previous Story Insights
- Story 4.1 built `SessionSummary` (label, archer_count, winner, winning_score), `archery_service._summarise` (tie-break: score desc, then `casefold()` name), and `HistoryListItem.vue` (subtext: `{count} archers — Winner: {winner} ({winning_score})`). This story extends the summary with top-3 and updates the subtext.
- Story 5.1 routes the label through `formatSessionLabel` in `HistoryListItem.vue`; Story 6.2 may show a custom `name`. Keep the label/name rendering as-is — this story only changes the **subtext** line, not the title line.

### Source references
- `backend/app/schemas/session.py` — `SessionSummary`.
- `backend/app/services/archery_service.py` — `_summarise`, `_archer_total`, `list_history`. Note `_archer_total` already treats missing archers as 0 (and per Story 7.1 will treat `null` shots as 0).
- `backend/app/routers/archery.py` — `GET /api/archery/sessions` → `list[SessionSummary]`.
- `frontend/src/apps/archery/types.ts` — `SessionSummary`.
- `frontend/src/apps/archery/components/HistoryListItem.vue` — subtext + aria-label.

### JSON field naming
Follow the snake_case convention (source-tree): `top_archers`, each `{ "name": ..., "score": ... }`. The frontend type mirrors this exactly (no camelCase remap in v1 — existing types already use `archer_count`, `winning_score`).

### Constraints / anti-patterns
- ❌ Don't compute standings on the frontend — the service owns ranking (consistent with FR-2.19 / Story 4.1).
- ❌ Don't drop `winner`/`winning_score` — other code/tests may rely on them.

### Testing
- **Backend:** pytest. **Frontend:** Vitest + `@vue/test-utils`. Update the existing `HistoryListItem.spec.ts` winner assertions to the new top-3 format.
