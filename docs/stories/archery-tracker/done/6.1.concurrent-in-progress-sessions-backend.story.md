## Story

**As a** developer,
**I want** the backend to persist and manage multiple in-progress sessions, each with a name and date,
**so that** operators can run several rounds at once and the frontend can list, resume, and finalise any of them.

## Acceptance Criteria

1. **Given** the `SessionData` schema, **When** a session is created, **Then** it includes a `name` (str, user-facing) and a `date` (str, `YYYY-MM-DD`) field in addition to the existing `label`; `label` remains the canonical unique id and filename stem.
2. **Given** `session_repo.py`, **When** an in-progress session is written, **Then** it is saved to its own file named `_ip_{label}.json` (underscore-prefixed so it is excluded from history listing), not a single shared `_in_progress.json`.
3. **Given** `session_repo.list_in_progress()`, **When** called, **Then** it returns all `_ip_*.json` sessions as `SessionData`, sorted newest first by `date` then label suffix.
4. **Given** `session_repo.read_in_progress(label)` and `delete_in_progress(label)`, **When** called with a label, **Then** they operate on that specific in-progress file; `delete_in_progress(label)` is idempotent.
5. **Given** `archery_service.create_session(...)`, **When** called, **Then** it no longer raises if another session is in progress â€” it always creates a new in-progress session with a fresh unique label, the supplied/derived name, and today's date.
6. **Given** `archery_service.finalise_in_progress(label)`, **When** called, **Then** it finalises the specified in-progress session to `{label}.json` and deletes that session's `_ip_{label}.json` (other in-progress sessions are untouched).
7. **Given** the routers, **When** the in-progress endpoints are called, **Then** they accept/return a label: `GET /api/archery/sessions/in-progress` returns an array of in-progress summaries; `GET|PUT|DELETE /api/archery/sessions/in-progress/{label}` and `POST /api/archery/sessions/in-progress/{label}/finalise` target a specific session.
8. **Given** a previously written single `_in_progress.json` file exists, **When** the backend lists in-progress sessions, **Then** it is migrated/treated as an `_ip_{label}.json` session (no data loss) â€” a one-time read-and-rewrite is acceptable.
9. **Given** all existing tests, **When** the suite runs, **Then** tests are updated to the new multi-session contract and pass.

## Dev Notes

### Previous Story Insights
- Stories 2.1, 2.5, 3.1, 3.2 built the single-session model: one `_in_progress.json`, `create_session` raises `RuntimeError("session already in progress")` (translated to HTTP 409 in `routers/archery.py`), and `finalise_in_progress()` takes no args. This story **removes the single-session constraint**.
- `_pick_finalise_label(base, prefer)` (in `archery_service.py`) already handles finalised-filename collisions and intentionally ignores the in-progress file â€” keep that behaviour, just scope deletion to the finalised session's own in-progress file.

### Source references
- Schema: `backend/app/schemas/session.py` â€” `SessionData`, `CreateSessionRequest`, `SessionStatus`.
- Repo: `backend/app/repositories/session_repo.py` â€” `_atomic_write_json`, `write_in_progress`, `read_in_progress`, `delete_in_progress`, `list_sessions`, `_SESSION_LABEL_RE`.

- Router: `backend/app/routers/archery.py` â€” current in-progress endpoints + `_LABEL_PATTERN`.
- Architecture: `docs/architecture/source-tree.md` (layering: routers â†’ services â†’ repositories; file I/O only in repo) and `tech-stack.md`.

### Key model decisions
- **`label` stays canonical.** Filenames, sort keys, path params all use `label` (`YYYY-MM-DD[-N]`). `name` is display-only; `date` is `YYYY-MM-DD` for ordering. This mirrors the storage/display split established in Story 5.1.
- **In-progress filename scheme:** `_ip_{label}.json`. The leading underscore guarantees `list_sessions()` (history) keeps excluding it. Do NOT reuse `{label}.json` for in-progress â€” that name is reserved for finalised sessions and would collide on finalise.
- **Layering:** all globbing/reading/writing stays in `session_repo.py`. The service decides *which* label; the repo decides *where on disk*.

### Constraints / anti-patterns
- âťŚ Do not put file I/O in the service or router.
- âťŚ Do not derive any filename from `name` (names are free text and non-unique).
- âťŚ Do not break the finalised-file format â€” finalised JSON must remain identical to Story 2.5 output (this story only changes in-progress storage + adds `name`/`date`).
- âš ď¸Ź This story changes the FEâ†”BE in-progress contract (see the BLOCKING note at the top and Task 6). Do not leave the FE calling the removed single-object `GET /sessions/in-progress`.
