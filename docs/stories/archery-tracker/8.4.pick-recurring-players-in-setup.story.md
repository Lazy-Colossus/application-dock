# Story 8.4: Pick Recurring Players in Session Setup

## Status

Draft

## Story

**As an** operator,
**I want** to add recurring players to a new session's roster from a picker,
**so that** I can build a roster quickly without retyping familiar names.

## Acceptance Criteria

1. **Given** the session setup screen, **When** it renders, **Then** the free-text custom-name input remains, plus a control to pick from recurring players.
2. **Given** the operator opens the recurring-players picker, **When** it loads, **Then** it lists recurring players not already in the draft roster.
3. **Given** the operator selects a recurring player, **When** the selection registers, **Then** that name is added to the draft roster (same uniqueness rules as typed names).
4. **Given** a recurring player already in the roster, **When** the picker renders, **Then** it is hidden or disabled to prevent duplicates.
5. **Given** no recurring players exist, **When** the picker is opened, **Then** an empty state points the operator to the management view (Story 8.3).

## Tasks / Subtasks

- [ ] **Task 1 — Load recurring players on setup** (AC: 1, 2)
  - [ ] In `SessionSetupPage.vue`, use `useRecurringPlayersStore` (Story 8.3) and call `loadPlayers()` on mount.
  - [ ] Add a "Add from recurring" control (e.g. a button opening a `q-menu`/`q-dialog`, or `q-select` of recurring names) next to the custom-name input.

- [ ] **Task 2 — Picker contents & dedup** (AC: 2, 4)
  - [ ] Compute `availablePlayers = recurringPlayers.filter(n => !store.draftRoster.includes(n))`.
  - [ ] Render only `availablePlayers` (already-rostered names hidden/disabled).

- [ ] **Task 3 — Add on select** (AC: 3)
  - [ ] On selecting a recurring name, push it to `store.draftRoster` using the same guard as `addArcher()` (trim, non-empty, not already present). Reuse/extract a small `addName(name)` helper so typed and picked names share validation.

- [ ] **Task 4 — Empty state** (AC: 5)
  - [ ] If `recurringPlayers.length === 0`, the picker shows "No recurring players — add some in Manage Players" with a link/button to `/archery/players` (Story 8.3).

- [ ] **Task 5 — Tests** (AC: 1–5)
  - [ ] `SessionSetupPage.spec.ts`: picker lists only not-yet-rostered recurring names; selecting adds to roster; rostered names excluded; empty state links to management.

## Dev Notes

### Previous Story Insights
- **Depends on Stories 8.2 (API) and 8.3 (`useRecurringPlayersStore`).** Reuses `store.draftRoster` and the `addArcher` validation from Story 2.2.
- This page also gains a "Session name" input (Story 6.2) and focus retention (Story 8.1). Keep the layout coherent: Session name → custom archer input + Add + "Add from recurring" → roster list → Confirm.
- Extract the name-add validation into a shared helper so the custom input, Enter key, and recurring picker all enforce identical rules (avoids divergence).

### Source references
- `frontend/src/apps/archery/pages/SessionSetupPage.vue` — `addArcher()`, `store.draftRoster`, roster list rendering.
- `frontend/src/apps/archery/stores/useRecurringPlayersStore.ts` — `players`, `loadPlayers()` (Story 8.3).
- `frontend/src/apps/archery/stores/useArcherySessionStore.ts` — `draftRoster`, `draftName`.
- `frontend/src/router/routes.ts` — `/archery/players` route (Story 8.3) for the empty-state link.

### UX note
The custom free-text field MUST remain (FR-8.3) — the picker augments, not replaces, it. A one-off guest archer can still be typed without becoming a recurring player.

### Constraints / anti-patterns
- ❌ Don't auto-add all recurring players — selection is explicit.
- ❌ Don't bypass roster uniqueness/trim rules for picked names.
- ❌ All HTTP via the store/`useApi.ts`.

### Testing
- **Frontend:** Vitest + `@vue/test-utils` + `@pinia/testing`. Provide a stubbed `useRecurringPlayersStore` with controllable `players`. Assert filtering against `draftRoster`, add-on-select, and empty-state behaviour.
