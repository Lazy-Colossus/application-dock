# Story 8.3: Recurring Players Management UI

## Status

Draft

## Story

**As an** operator,
**I want** to manage my list of recurring players from the Archery home screen,
**so that** I can curate the names that appear when building a roster.

## Acceptance Criteria

1. **Given** the Archery home screen, **When** it renders, **Then** there is an entry point (button) to a recurring-players management view.
2. **Given** the management view, **When** it loads, **Then** it lists current recurring players (from `GET /api/archery/players`) with a way to remove each (≥44px targets).
3. **Given** the management view, **When** the operator enters a name and adds it, **Then** `POST /api/archery/players` is called and the list updates; empty/duplicate handled with feedback.
4. **Given** the operator removes a player, **When** confirmed, **Then** `DELETE /api/archery/players/{name}` is called and the list updates.
5. **Given** a dedicated Pinia store (`useRecurringPlayersStore`), **When** actions run, **Then** they expose `loading`/`error` and use a `finally` block; all HTTP goes through `useApi.ts`.

## Tasks / Subtasks

- [ ] **Task 1 — Pinia store** (AC: 5)
  - [ ] Create `frontend/src/apps/archery/stores/useRecurringPlayersStore.ts` with `players` ref, `loading`, `error`, and actions `loadPlayers()`, `addPlayer(name)`, `removePlayer(name)`.
  - [ ] All actions set `loading=true`, clear `error`, use `try/catch/finally`, and call through `@/composables/useApi` (`api.get`/`api.post`/`api.del`).

- [ ] **Task 2 — Management page + route** (AC: 1, 2)
  - [ ] Create `frontend/src/apps/archery/pages/RecurringPlayersPage.vue`.
  - [ ] Add a lazy-loaded route in `frontend/src/router/routes.ts`, e.g. `archery/players` → `RecurringPlayersPage.vue`, `meta: { title: 'Players' }`.
  - [ ] On `ArcheryHomePage.vue`, add an entry-point button (e.g. "Recurring Players" / "Manage Players") routing to `/archery/players`. Keep it visually subordinate to New Session/Resume/History (Story 6.3).

- [ ] **Task 3 — List + add + remove UI** (AC: 2, 3, 4)
  - [ ] On load (`onMounted`), call `store.loadPlayers()`.
  - [ ] Render the list (reuse `ArcherChip.vue` with `:removable="true"` and a `@remove` handler, or a `q-list`); each row/chip ≥44px.
  - [ ] Add a name input + Add button (mirror `SessionSetupPage` add pattern, including focus-retention from Story 8.1 if convenient). Empty/duplicate → inline error.
  - [ ] Remove → `store.removePlayer(name)` (optionally a confirm for safety; a single tap-to-remove chip X is acceptable since the list is non-destructive to sessions).
  - [ ] Empty state: "No recurring players yet — add names to reuse them across sessions."

- [ ] **Task 4 — Tests** (AC: 2–5)
  - [ ] `useRecurringPlayersStore.spec.ts`: load/add/remove call the right `useApi` methods; `loading` toggles; `error` set on failure.
  - [ ] `RecurringPlayersPage.spec.ts`: lists players; add posts; remove deletes; empty state renders.

## Dev Notes

### Previous Story Insights
- **Depends on Story 8.2** (the `/api/archery/players` endpoints). **Coordinates with Story 6.3** (home redesign) for placement of the entry-point button — add it as a secondary control on `ArcheryHomePage.vue` alongside New Session / Resume / History.
- `ArcherChip.vue` (Story 2.2) already supports `:removable` + `@remove` — reuse it for the players list.
- Store conventions (from `useArcherySessionStore`/`useArcheryHistoryStore`): expose `loading`/`error`, `finally` block, HTTP only via `useApi.ts`. Stores never call other stores.

### Source references
- `frontend/src/apps/archery/stores/useArcheryHistoryStore.ts` — pattern for a read-mostly store.
- `frontend/src/apps/archery/components/ArcherChip.vue` — removable chip.
- `frontend/src/apps/archery/pages/SessionSetupPage.vue` — add-name input pattern (and Story 8.1 focus retention).
- `frontend/src/router/routes.ts` — lazy route registration pattern.
- `frontend/src/composables/useApi.ts` — `api.get/post/del`, throws `ApiError`.
- Architecture: `docs/architecture/source-tree.md` — `apps/archery/` is self-contained; stores in `stores/`, pages in `pages/`.

### Constraints / anti-patterns
- ❌ All HTTP via `useApi.ts`; no raw fetch/axios.
- ❌ Don't import shell/other-app code; stay within `apps/archery/`.
- ❌ Don't reuse `useArcherySessionStore` for players — separate concern, separate store.

### Testing
- **Frontend:** Vitest + `@vue/test-utils` + `@pinia/testing`. Stub `useApi`. Co-located specs.
