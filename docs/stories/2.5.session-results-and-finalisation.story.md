# Story 2.5: Session Results & Finalisation

## Status

Ready for Review

## Story

**As an** operator,
**I want** to view ranked results with a per-target breakdown and finalise the session to permanent storage,
**so that** the session outcome is recorded and the group can see who won.

## Acceptance Criteria

1. **Given** all 18 targets are confirmed (all green), **When** the scoring board renders, **Then** a non-blocking "View Results" prompt appears; the operator can still tap any target to revisit it.
2. **Given** the operator navigates to the results screen, **When** it renders, **Then** each archer is shown with their total score (sum of all shots across all targets; max 396), ranked highest to lowest.
3. **Given** the results screen, **When** the operator scrolls, **Then** a per-archer, per-target table is visible showing: target number, shot 1, shot 2, subtotal for each archer.
4. **Given** the results screen, **When** the operator taps "Return to Scoring", **Then** they return to the scoring board with all target states intact.
5. **Given** the operator taps "Finalise Session", **When** the action is confirmed, **Then** `POST /api/archery/sessions/in-progress/finalise` is called; the session is written to `DATA_DIR/YYYY-MM-DD.json` (or `YYYY-MM-DD-2.json` etc.) with `status: finalised`; the operator is returned to the Archery home screen.
6. **Given** the finalise endpoint is called, **When** a file with the same label already exists, **Then** the next available suffix is used (e.g. `-2`, `-3`) and the file is written atomically.

    ```ts
    async function finaliseSession() {
      if (!session.value) return
      loading.value = true
      error.value = null
      try {
        const finalised = await api.post<SessionData>('/archery/sessions/in-progress/finalise', null)
        // server returns the finalised session (status: 'finalised', label maybe changed if collision)
        session.value = null   // clear active session — it's now historical
        // optional: notify history store to refresh on next mount
      } catch (e) {
        error.value = e instanceof Error ? e.message : String(e)
        throw e   // let the page decide what to do
      } finally {
        loading.value = false
      }
    }
    ```
  - [x] Also add a totals helper (used by `ResultsPage` and `ResultsTable`):
    ```ts
    function totalFor(session: SessionData, archer: string): number {
      return session.targets.reduce((sum, t) => sum + (t.scores[archer]?.[0] ?? 0) + (t.scores[archer]?.[1] ?? 0), 0)
    }
    function ranked(session: SessionData): Array<{archer: string, total: number}> {
      // Tie-break: alphabetical archer name (case-insensitive) — mirrors backend rule (Story 4.1 v0.2)
      return session.archers
        .map(a => ({ archer: a, total: totalFor(session, a) }))
        .sort((x, y) => y.total - x.total || x.archer.localeCompare(y.archer, undefined, { sensitivity: 'base' }))
    }
    ```

### API contract — `POST /api/archery/sessions/in-progress/finalise`
[Source: architecture.md#API & Communication Patterns]

- **Request body:** none (empty POST; the server reads `_in_progress.json` to know what to finalise)
- **Success (200 OK):** `SessionData` with `status: "finalised"` and the finalised `label` (which may differ from the in-progress label if a collision forced a suffix bump)
- **404 Not Found:** No `_in_progress.json` exists. `{"detail": "No session in progress."}`
- **No envelope.** Direct serialization.
- **Atomicity:** file write via `_atomic_write_json` (Story 2.1) — `.tmp` → `os.replace()` [Source: architecture.md#Enforcement Guidelines]

### Score totals — math reference (FR-2.14)
- Max total = 18 targets × 2 shots × 11 = 396
- Per-archer total = sum of all shot values across all confirmed targets for that archer
- Per-archer per-target subtotal = `shot1 + shot2` (max 22)
- Ranking: descending by total, ties broken arbitrarily (no tiebreaker rule in v1; preserve roster order on ties is acceptable)

### Constraints / anti-patterns
- ❌ Do not compute totals on the backend for the live results view — the FE already has `session.targets` in the store; computing client-side avoids an extra round-trip [Source: efficient API design — local data already available]
- ❌ Do not allow editing after finalise (FR-2.21); history detail (Story 4.2) MUST be read-only — flag this here so reviewers catch any leak between live results and history detail components (they share `ResultsTable.vue` but the action buttons differ)
- ❌ Do not write the finalised file before the rename — always go through `_atomic_write_json` [Source: architecture.md#Enforcement Guidelines]
- ❌ Do not envelope the response [Source: architecture.md#Anti-Patterns to Avoid]
- ❌ Do not navigate to history immediately after finalise — return to Archery home per FR-2.17
