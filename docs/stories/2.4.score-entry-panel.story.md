# Story 2.4: Score Entry Panel

## Status

Ready for Review

## Story

**As an** operator,
**I want** to enter two shots per archer for a target using large tap buttons,
**so that** I can record scores quickly with one hand while outdoors.

## Acceptance Criteria

1. **Given** the score-entry panel opens for a target, **When** it renders, **Then** it shows the first archer's name and two shot slots; valid shot buttons (0, 5, 8, 10, 11) are displayed, each ≥44px.
2. **Given** the operator taps a shot value for the first shot, **When** the value is selected, **Then** the first shot slot is filled and the second shot slot becomes active.
3. **Given** both shots for an archer are entered, **When** the second shot is selected, **Then** the panel automatically advances to the next archer in roster order.
4. **Given** the 11 button, **When** it renders, **Then** it is visually distinguished with a gold highlight distinct from the other shot buttons.
5. **Given** the operator is on any archer (not the first), **When** they tap "Back", **Then** they return to the previous archer's entry with their previously entered values preserved.
6. **Given** all archers have both shots entered, **When** the operator taps "Confirm Target", **Then** the panel closes, the target icon turns green on the board, and the session store is updated.
7. **Given** the operator re-opens a confirmed target, **When** the score-entry panel renders, **Then** the previously entered shot values are pre-populated for each archer; the target's green state is reset until re-confirmed.

## Dev Notes

### Previous Story Insights
- Story 2.1: `SessionData`, `TargetScores` schemas. `TargetScores.scores` is `Record<string, [int, int]>` — both ints required (no partial state on the wire). For an in-progress target (only one shot entered, or some archers incomplete), the entry stays out of `session.targets` until "Confirm Target" — entries with partial pairs do NOT enter the persisted session.
- Story 2.2: `useArcherySessionStore` with `session: ref<SessionData | null>`. Backend creates session via `POST /api/archery/sessions`, which sets `_in_progress.json`. Local store has the `session` ref hydrated from that response.
- Story 2.3: `TargetIcon.vue`, `ScoringBoardPage.vue`, `ArcherChip.vue`. Store has `activeTargetNumber`, `scoreEntryOpen`, `openTarget`, `closeTarget`, `targetByNumber`, `isConfirmed`. Reuse these.

### Behavioral spec
[Source: EXPERIENCE.md#Interaction Primitives, #State Patterns, #Key Flows — Flow 1]

- **Auto-advance:** After the second shot button tap for an archer, panel auto-advances to next archer (no tap required) — AC 3
- **Back behavior:** Steps to previous archer (or previous shot slot within current archer); preserves values; operator can tap a different shot button to overwrite (AC 5)
- **Confirm Target enabled:** Only when EVERY archer has both shots filled (no nulls)
- **Reduce motion:** Skip slide-up animation [Source: EXPERIENCE.md#Accessibility Floor]
- **Focus trap:** Focus stays within sheet while open; first shot button receives focus on render [Source: EXPERIENCE.md#Accessibility Floor]
- **Backdrop dismiss:** Blocked if any shot entered for current open (prevents accidental data loss)
- **No special "partial discard" UX in v1** — backing all the way out without confirming simply leaves the target unconfirmed; entries are local to the panel and not committed until Confirm Target. (EXPERIENCE.md mentions "Discard entry?" confirmation for partial entry — implement this as a guard on closeTarget() if any non-null entries exist: show a `q-dialog` "Discard entry?" with Cancel / Discard before actually closing.)

### Constraints / anti-patterns
- ❌ Do not validate shot values in the FE only — values are typed `0 | 5 | 8 | 10 | 11` (TS union) and the backend's Pydantic validator enforces the same; if a non-valid value somehow reaches the API, Pydantic will return 422 [Source: Story 2.1 schemas]
- ❌ Do not store partial entries in `session.targets` — a target only enters the array on Confirm Target [Source: architecture.md#Data Architecture]
- ❌ Do not place shot buttons or panel under `src/components/` — they're archery-specific [Source: architecture.md#Anti-Patterns to Avoid]
- ❌ Do not introduce a separate Pinia store for panel state — keep local component state OR co-locate refs in `useArcherySessionStore`. The chosen pattern in this story is component-local state for the in-flight panel session.
- ❌ Do not call `fetch` in the component — go through the store, which goes through `useApi` [Source: architecture.md#Enforcement Guidelines]
