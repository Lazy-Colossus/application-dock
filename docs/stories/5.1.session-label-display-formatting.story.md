# Story 5.1: Session Label Display Formatting

## Status

Ready for Review

## Story

**As an** operator,
**I want** session labels to be displayed as `YYYY-MM-DD #N` instead of `YYYY-MM-DD-N` wherever they appear in the UI,
**so that** the user-facing label matches the format described in the PRD (FR-2.4) and reads naturally as "second session of the day."

## Acceptance Criteria

1. **Given** a session has the stored label `2026-05-28-2`, **When** that label is rendered anywhere in the UI, **Then** it is displayed as `2026-05-28 #2`.
2. **Given** a session has the stored label `2026-05-28` (first session of the day, no suffix), **When** that label is rendered anywhere in the UI, **Then** it is displayed unchanged as `2026-05-28`.
3. **Given** the display transform, **When** it is invoked, **Then** the underlying stored label (the canonical value on disk and in API payloads) is **never** modified — the transform is presentation-only.
4. **Given** the History list, **When** a row renders, **Then** the label shown follows the `YYYY-MM-DD #N` convention (covers `HistoryListItem.vue`).
5. **Given** the History detail page, **When** the session heading renders, **Then** the label shown follows the `YYYY-MM-DD #N` convention (covers `HistoryDetailPage.vue`).
6. **Given** the Results page (live session), **When** the session label heading renders, **Then** it follows the `YYYY-MM-DD #N` convention (covers `ResultsPage.vue`).
7. **Given** the Resume bottom sheet on Archery Home, **When** the in-progress session sub-line renders, **Then** the label shown follows the `YYYY-MM-DD #N` convention (covers `ArcheryHomePage.vue` resume sheet from Story 3.2).
8. **Given** any other surface that renders a session label (now or in the future), **When** it does so, **Then** it uses the same `useSessionLabel` composable — no inline string manipulation.

## Tasks / Subtasks

- [ ] **Task 1 — `useSessionLabel.ts` composable** (AC: 1, 2, 3, 8)
  - [ ] Create `frontend/src/apps/archery/composables/useSessionLabel.ts`
  - [ ] Export a pure function `formatSessionLabel(label: string): string`:
    ```ts
    const SUFFIX_RE = /^(\d{4}-\d{2}-\d{2})-(\d+)$/
    export function formatSessionLabel(label: string): string {
      const m = label.match(SUFFIX_RE)
      return m ? `${m[1]} #${m[2]}` : label
    }
    ```
  - [ ] Export it from the composable as both a plain function and a `useSessionLabel()` composable that returns `{ format: formatSessionLabel }` — call sites can use either form.

- Epics: `docs/epics.md` FR-2.4 — same display form

### The split between storage and display

The CANONICAL form is `YYYY-MM-DD[-N]` (used in filenames, JSON `label` field, API path params, sort keys). This story does NOT change any of that.

The PRESENTATIONAL form is `YYYY-MM-DD [#N]`. This story introduces a single transform function and routes all rendering through it.

**Do not change** any of:
- Backend `archery_service.generate_session_label()` output
- Backend file naming
- API path param format on `GET /api/archery/sessions/{label}` (still `2026-05-28-2`, not `2026-05-28 #2`)
- Stored `session.label` field
- Frontend sort keys, dictionary keys, or any non-display use

### Why a composable for a one-line function

Convention. Project standard puts shared display formatting in `composables/`. A future change (e.g. switching `#N` to `(N)`, or localising) is a one-place edit. Inline string manipulation in components would scatter the rule.

### Constraints / anti-patterns
- ❌ Do not mutate stored labels at the API or repository level.
- ❌ Do not write inline regex / split logic in components — always use `formatSessionLabel`.
- ❌ Do not localise the `#` symbol in v1 — there is no i18n layer; if i18n arrives later this composable is the seam.
- ❌ Do not change the path param format on history detail GET — that would break URL bookmarks.
