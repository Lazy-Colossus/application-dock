## Decisions

1. **Two router files (`router/index.ts` + `router/routes.ts`).** Quasar v2 / @quasar/app-vite v2 expects this split — `index.ts` is a Quasar `route()` wrapper that builds the router; `routes.ts` exports the route list. The story specifies edits to "`src/router/index.ts`" but the practical edit is to `routes.ts`. Story 1.1 already created both — this story modifies `routes.ts` to add `/archery`.

2. **`MainLayout` wraps every route.** In Story 1.1 the layout is a child of `/` (parent route in `routes.ts`). For the persistent home button + back arrow to appear on EVERY screen in every app (FR-1.3), I'll change the router structure so `MainLayout` is the parent of all top-level routes, not just `/`. The 404 page can either bypass the layout or live inside it; I'll keep it inside for consistency (the home button on a 404 is useful).

3. **`AppCard` is a `<router-link>` wrapping the whole card.** The story spec says "tap → navigate to app route". Wrapping in `<router-link>` makes the entire card a click target with proper a11y (focusable, keyboard-activatable). The card uses `<q-card>` for visual styling but the actual link is the wrapper.

4. **`useApi.ts` — minimum viable implementation.** Native `fetch` (no axios dep needed; smaller bundle). One module, four exports (`get/post/put/del`), one `ApiError` class. Network failures throw `ApiError(0, 'Network error')` per the v0.2 contract. JSON-parsing failure throws `ApiError(status, response.statusText, raw)`.

5. **Backend `GET /api/apps`** returns a hard-coded list matching `registry.ts`. They're independent sources that happen to agree in v1 — the architecture document is explicit on this. I'll define `AppDescriptor` as a Pydantic model inline in `routers/shell.py` (acceptable per story).

6. **Frontend tests — minimal Vitest setup.** Add `vitest.config.ts` aliased like Quasar (resolve `@` → `src/`), plus a `setup.ts` that registers Quasar via `installQuasarPlugin` from `@quasar/quasar-app-extension-testing-unit-vitest` — actually that's a Quasar AE that's overkill here. Simpler: bare Vitest + happy-dom; mount Quasar via `@vue/test-utils` `global.plugins` with a minimal `Quasar` install. Test the AppCard renders correctly and emits navigation; skip full Quasar plugin integration for now.

7. **Carbon theme verification.** Mockup `shell-home.html` uses pure-color SVG icons; Quasar Material Icons (`sports_score`) is a font icon. For v1 the icon name in `registry.ts` is a Material Icons identifier — `<q-icon name="sports_score" color="primary" size="32px" />` honors the accent color.

## Risk register

| Risk | Mitigation |
|------|------------|
| Quasar in test env doesn't auto-register components (`q-icon`, etc.) — tests fail with "Failed to resolve component" | Use `mount(..., { global: { stubs: { 'q-icon': true } } })` or install the Quasar plugin in test setup. Prefer minimal stubs to keep tests fast. |
| Router-link inside `q-card` traps clicks awkwardly | Use `<q-card clickable @click="..."` with `router.push` OR wrap in `<router-link custom>` with a slot. I'll pick `<router-link>` wrapping the card for simplest semantics. |
| 360px viewport on Quasar grid wraps unexpectedly | Use plain CSS Grid (`grid-template-columns: 1fr 1fr`) with `gap: 12px` — avoids Quasar's column system complexity at this scale. |
| Accent color in icon doesn't render as gold (#C8960A) | quasar.variables.sass + framework.config.brand.primary both set to `#C8960A`; verify in dev. |
| Story spec asks me to register a `frontend/src/css/quasar.variables.sass` Sass include — already created by 1.1 | Reuse the existing file. |
