# Story 1.4: Settings ‚Äî "Update applications" Button

## Status

review

## Story

**As an** operator,
**I want** an "Update applications" button on the Settings page that triggers the host's update script (`/scripts/update-application-dock-docker.sh`),
**so that** I can pull the latest images and restart the running stack without dropping to a host shell ‚Äî and so the button gracefully tells me "Update not available" when no script is installed on the host.

## Acceptance Criteria

1. **Given** the host has a script at `/scripts/update-application-dock-docker.sh` and the operator's `docker-compose.yml` mounts that directory into the container (per the new compose contract in Task 1), **When** the backend boots, **Then** the existence of that script is checked **once at startup** and cached as `update_available: bool` in `app/services/update_service.py`. The check is not re-performed on subsequent requests.
2. **Given** the cached `update_available` flag, **When** the frontend calls `GET /api/shell/update-status`, **Then** the response is `{ "available": <bool> }` (snake_case JSON, direct serialization, per CLAUDE.md API contract).
3. **Given** the Settings page mounts and the API returns `{ available: false }`, **When** the page renders, **Then** the "Update applications" button is **disabled**, its label reads `Update not available`, and a short tooltip / helper line explains "No update script found on host". No POST is possible.
4. **Given** the API returns `{ available: true }`, **When** the page renders, **Then** the button is **enabled** with the label `Update applications`. Tapping it opens a Quasar confirm dialog whose body reads: *"This will pull the latest images and restart the running container. The page will lose connection briefly ‚Äî refresh once it's back."* Confirm and Cancel actions only.
5. **Given** the operator confirms the dialog, **When** the SPA posts to `POST /api/shell/update`, **Then** the backend (a) re-validates `update_available` (returns `503` with `{ "detail": "Update not available" }` if false), (b) uses the Docker SDK to spawn a sidecar container per the contract in Task 2, (c) returns `202 Accepted` with `{ "detail": "Update started" }` immediately ‚Äî **before** the sidecar runs the script (fire-and-forget). The backend does **not** wait for the script's output, exit code, or completion.
6. **Given** the `202` response, **When** the SPA receives it, **Then** it shows a non-blocking toast/banner ("Update started ‚Äî connection may drop") and disables the button. No polling, no version-check, no auto-reload ‚Äî the operator refreshes the page manually once the new container is up (matches the "fire-and-forget + warning" UX choice).
7. **Given** the operator's `docker-compose.yml` does **not** mount the docker socket or the scripts directory, **When** the backend boots, **Then** `update_available` is `false` (the existence check fails cleanly) and the Settings page renders the disabled state ‚Äî i.e. the feature gracefully degrades when the operator hasn't opted into the new compose contract. No exception, no startup failure.
8. **Given** the existing `SettingsPage.vue` stub from Story 1.3, **When** this story lands, **Then** the empty-state placeholder is replaced with a real Settings page that has a single section labelled `MAINTENANCE` containing the Update button. The page structure remains room-to-grow (a future story will add more sections).

### Architectural rationale (why docker.sock + sidecar)

We chose the docker-socket-mounted sidecar pattern over alternatives (host trigger file, host HTTP service) because:
- **Single setup step** for the operator (two mounts in `docker-compose.yml`) vs. installing a host-side daemon.
- **No new long-running host process** to keep alive; the sidecar exits when the script does.
- **Reuses an existing trust boundary** ‚Äî the operator already trusts whoever wrote `docker-compose.yml` to do as they please on the host.

The cost is **explicit and documented**: mounting the docker socket gives the app container root-equivalent access to the host. This is acceptable for the project's stated scope (single-user home use; "No auth, no CORS, no DB" per the v1 architecture). If the threat model ever changes (multi-tenant, untrusted network), this feature **must** be removed or replaced ‚Äî flag this in the deferred-work log if you encounter scope drift.

### The dual-env-var oddity

When the backend asks Docker to "mount the host's `/scripts` into the sidecar," the Docker API requires the **host path**, not the in-container path. The backend itself only sees `/host-scripts` (its own mounted view). So we need two env vars:
- `HOST_SCRIPTS_DIR` ‚Äî the host path the compose file uses on its `volumes:` line (e.g. `/opt/application-dock/scripts`)
- `HOST_SCRIPTS_DIR_ON_HOST` ‚Äî the same value, passed into the container as an env var, so the backend can use it as the **source** of the sidecar's bind mount

In practice, the operator sets both to the same value (or compose interpolates: `environment: HOST_SCRIPTS_DIR_ON_HOST=${HOST_SCRIPTS_DIR}`). This is the unavoidable awkwardness of the docker.sock pattern ‚Äî the container is asking the daemon to mount a path the daemon sees, not a path the container sees. Document it; do not paper over it.

### Constraints / anti-patterns
- ‚ùå Do **not** wait for the sidecar to finish. Fire-and-forget per AC 5. Streaming logs or polling exit codes will block the request and likely fail when the parent container is restarted by the script.
- ‚ùå Do **not** add a "What does the script do?" preview that runs `head -n ‚Ä¶` on the script. The script is an opaque operator hook; surfacing its contents creates a false sense of safety.
- ‚ùå Do **not** add auth on `POST /api/shell/update`. v1 has no auth anywhere (per architecture); adding it just here would be inconsistent and would not actually protect anything if the network is already trusted to talk to `/api/archery/*`.
- ‚ùå Do **not** make `HOST_SCRIPTS_MOUNT` configurable at runtime via env var. The in-container path is a **compose contract** ‚Äî making it configurable invites operators to misconfigure the mount and the existence check independently. One source of truth. **This is enforced in code**: `HOST_SCRIPTS_MOUNT` lives as a hardcoded constant in `update_service.py`, not in `Settings` (where `pydantic-settings` would silently honour an `HOST_SCRIPTS_MOUNT=‚Ä¶` env var).
- ‚ùå Do **not** auto-pull `docker:cli` from the backend. If the image is missing, the launch fails and the operator sees a 502 ‚Äî that's the right signal. Auto-pulling hides the dependency and slows the first invocation by ~10s.
- ‚ùå Do **not** add a Pinia store for this. Local refs in `SettingsPage.vue` are enough; there's no cross-component state to share.
- ‚úÖ The `application-dock-updater` name on the sidecar is **intentional** ‚Äî if a previous run got stuck, the next launch will fail with "container name already in use", which is loud and correct. Do not silently `rm -f` a leftover container; that masks bugs.
- ‚úÖ When `available` is `false`, the button **must** be rendered (not omitted), so the feature's existence is discoverable. The disabled state + tooltip teach the operator how to turn it on.

### Security boundary (call it out for review)

This story crosses a security boundary by giving the backend the ability to spawn arbitrary containers on the host. The mitigations baked in:
- Only one specific image (`docker:cli`) and one specific command (`sh /host-scripts/update-application-dock-docker.sh`) are ever invoked.
- The mount is **read-only** on the script directory.
- No user input flows into the docker command ‚Äî the path, image, and command are all constants in `update_service.py`.

Anything that introduces user-controlled values (script name, arguments, image tag) **must** be flagged in code review and escalated.

### Testing
- **Backend:** pytest with `unittest.mock.MagicMock` for the Docker client; never touch a real daemon in tests. The `is_update_available()` test must use `monkeypatch` + `importlib.reload` (or refactor the module to read the path lazily ‚Äî dev's call, but the test must not depend on the in-process boot state).
- **Frontend:** Vitest. Mock `useApi` per the existing idiom. Use Quasar's test plugins to mount `q-dialog` (see Quasar docs); if that's painful, stub the dialog and assert the click handler is wired correctly instead of asserting the dialog DOM ‚Äî the value is the wiring, not the framework.
- **Integration:** out of scope for this story. A future story can add a real end-to-end test that mounts `dind` and runs the full sidecar dance.

### Out of scope
- Authentication / authorization on the update endpoint.
- Streaming script output back to the UI.
- Auto-reload of the SPA after the new container boots.
- A "version" / "what's installed" panel on Settings (different story).
- Rollback after a failed update.
- Pre-pulling `docker:cli` from the backend.
- Distributing or version-controlling the host's update script.
