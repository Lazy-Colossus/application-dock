# Story 1.3: Toolbar Home Button Becomes Settings on the Main Screen

## Status

review

## Story

**As an** operator,
**I want** the toolbar's top-right control to be a Settings button when I am on the apps landing page (and a Home button on every other screen),
**so that** I can reach a Settings area from the main screen — where a Home button is redundant — without losing the persistent way home from inside any app.

## Acceptance Criteria

1. **Given** the operator is on the apps landing page (`/`), **When** the toolbar renders, **Then** the top-right control is a Settings button (Material icon `settings`, `aria-label="Open settings"`) that routes to `/settings`. The Home button does **not** render on this route.
2. **Given** the operator is on any non-root route (any `/archery/*` route, the new `/settings` route, or the not-found page rendered under `MainLayout`), **When** the toolbar renders, **Then** the top-right control is the existing Home button (Material icon `home`, `aria-label="Go to apps home"`, `to="/"`) — unchanged behaviour from today.
3. **Given** the operator taps the Settings button on the landing page, **When** the route change completes, **Then** the URL is `/settings`, the page is rendered through `MainLayout` (so the toolbar still appears, now showing the Home button per AC 2), the back arrow appears (per the existing `showBack = route.path !== '/'` rule), and the page title in the toolbar reads `Settings`.
4. **Given** the operator is on `/settings`, **When** the page renders, **Then** a new `SettingsPage.vue` mounts and displays an empty-state placeholder — a single heading or message such as `Settings` / `Nothing here yet`. No interactive settings controls are added in this story; the page is intentionally a stub for a later story to populate.
5. **Given** the existing persistent-home requirement (FR-1.3) and Story 1.2 AC 3 ("a persistent home control is visible … from every screen inside any app"), **When** an operator is inside any app screen, **Then** the Home button is still visible and functional — this story only **removes** Home from the landing page itself (where there is no "inside an app" to return from) and **adds** Settings there in its place.

## Dev Notes

### Previous Story Insights
- **Story 1.2** established the persistent Home button in the toolbar (`MainLayout.vue`) with `aria-label="Go to apps home"` and `to="/"`. That story's AC 3 specifically scopes the requirement to "every screen **inside any app**" — i.e. not the landing page itself. This story aligns the implementation with that scope by removing Home from `/` and using the freed slot for Settings.
- **Story 1.1** set the precedent that shell-owned pages live under `frontend/src/pages/` (e.g. `HomePage.vue`, `ErrorNotFound.vue`), while app-owned pages live under `frontend/src/apps/{app-id}/pages/`. The new `SettingsPage.vue` is shell-owned and belongs under `frontend/src/pages/`.

### Source references
- `frontend/src/layouts/MainLayout.vue` — the only file that needs to change to satisfy ACs 1–2. The current Home button is lines 18–26; the back arrow + title logic above is untouched.
- `frontend/src/router/routes.ts` — add the new child route under the existing `MainLayout` parent (lines 4–57); do **not** create a second top-level layout.
- `frontend/src/pages/HomePage.vue` — reference only, for the `.section-label` styling vocabulary the new `SettingsPage.vue` should mirror.
- `frontend/src/components/AppCard.spec.ts` — reference only, for the existing `RouterLinkStub` testing idiom.
- `docs/planning-artifacts/epics.md` — Story 1.2 AC 3 (line ~204) and FR-1.3 (line ~22) define the persistent-home scope; both are compatible with this story.

### Existing file state (read these before editing)

`MainLayout.vue` currently renders an unconditional Home `q-btn` and a conditional back arrow gated on `showBack = route.path !== '/'`. The new conditional must mirror that same boolean (inverted: `isHome = route.path === '/'`) so the two checks agree on a single source of truth (the route path). Do **not** introduce a `useRouter().currentRoute.value.name === 'home'` variant — `route.path` is what the existing code uses.

`routes.ts` already groups all shell routes under one `MainLayout` parent with `path: '/'`. Add the new `settings` route as a child of that same parent — do **not** create a sibling top-level route, or the toolbar will not render on `/settings` and AC 3 will fail.

### Constraints / anti-patterns
- ❌ Do **not** put the Settings button inside `HomePage.vue`. It must live in `MainLayout.vue`'s toolbar so it occupies the same screen slot as the Home button it replaces. Putting it in `HomePage.vue` would make the toolbar slot empty on `/` and force the Settings control into the page body — a different UX.
- ❌ Do **not** add a `meta: { showSettings: true }` flag or any per-route configuration to drive the swap. The condition is exactly one route (`/`), and a route-meta indirection would be premature abstraction (YAGNI per `CLAUDE.md`).
- ❌ Do **not** populate the `SettingsPage` with placeholder controls (theme toggle, "coming soon" tile grid, etc.). The user has explicitly scoped this story to a stub; content lands in a later story.
- ❌ Do **not** change the Home button's icon, colour, or `aria-label` on non-root routes — AC 2 is "unchanged from today" and any drift will break future shell-level tests.
- ✅ The page title for `/settings` comes from the existing `pageTitle = route.meta?.title` computed — set `meta: { title: 'Settings' }` on the new route and the toolbar title falls out for free. No template change needed for that.

### Material icon choice
Use `settings` (the gear icon) — it is the de-facto Material Symbols icon for app-level settings and matches the existing icon vocabulary (`home`, `arrow_back_ios_new`, `chevron_right`, `sports_score`). No other candidate (`tune`, `manage_accounts`, `build`) is appropriate here.

### Testing
- **Frontend:** Vitest + `@vue/test-utils`. For `MainLayout.spec.ts`, mount with a stubbed router (use `createRouter({ history: createMemoryHistory(), routes: [...] })` or set `route.path` via a test-utils `mocks` block — whichever the codebase already favours). For `SettingsPage.spec.ts`, a single shallow-mount smoke test is enough.
- **Backend:** none — this story is frontend-only. Do not touch `app/routers/shell.py`; the `_APPS` registry is for app cards, and Settings is a shell-level page, not an app.

### Out of scope
- Settings page content (a later story).
- Any theme / preferences / data-dir control surface.
- Mobile-only or desktop-only variants of the toolbar.
- A keyboard shortcut for Settings.
