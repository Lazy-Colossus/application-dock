## Story

**As a** developer,
**I want** a fully scaffolded project with a working dev environment and a single `docker compose up` production build,
**so that** the team can start building features immediately without environment friction.

## Acceptance Criteria

1. **Given** the repository is cloned, **When** `docker compose up` is run, **Then** the application is accessible at `http://localhost:8123` with a 200 response. *(Host port remapped to 8123 in `docker-compose.yml` вЂ” port 8000 was occupied on the dev host.)*
2. **Given** the dev environment is set up, **When** `npm run dev` (frontend) and `uvicorn app.main:app --reload` (backend) are started, **Then** the Quasar dev server runs on port 9000 and proxies `/api/*` to port 8000.
3. **Given** the project structure, **When** a developer inspects it, **Then** it matches exactly: `backend/app/{main.py,routers/,services/,repositories/,schemas/,core/}` and `frontend/src/{apps/,components/,layouts/,pages/,router/,composables/,css/}`.
4. **Given** the Docker build, **When** the image is built, **Then** a multi-stage Dockerfile runs `quasar build` in a node stage and copies `dist/spa/` into the Python stage; `requirements.txt` and `package.json` are the only dependency manifests.
5. **Given** the environment configuration, **When** `DATA_DIR` env var is set, **Then** it is read by `app/core/config.py`; if unset, it defaults to `/data` (and the directory is created on startup if it does not exist).

### Naming / patterns to bake in from day one
- API endpoints: `kebab-case` segments, plural вЂ” e.g. `/api/archery/sessions` [Source: architecture.md#Naming Patterns]
- JSON fields: `snake_case` everywhere [Source: architecture.md#Naming Patterns]
- Python: `snake_case` functions/vars, `PascalCase` classes, `snake_case` modules [Source: architecture.md#Naming Patterns]
- Vue components: `PascalCase` filename and name [Source: architecture.md#Naming Patterns]
- Pinia stores: `useXStore` camelCase [Source: architecture.md#Naming Patterns]
- API responses: **no envelope** вЂ” return Pydantic model or array directly [Source: architecture.md#Format Patterns, anti-patterns list]
- Dates: ISO 8601 strings in JSON [Source: architecture.md#Format Patterns]
- All HTTP from FE goes through `src/composables/useApi.ts` вЂ” never raw `fetch`/`axios` in components or stores [Source: architecture.md#Enforcement Guidelines]
- Atomic write pattern (`write .tmp в†’ os.replace()`) is the mandatory persistence pattern (not implemented in this story, but the directory `repositories/` exists for it) [Source: architecture.md#Enforcement Guidelines]

### Configuration
- `DATA_DIR` env var, default `/data` inside container [Source: architecture.md#Data Architecture, #Infrastructure & Deployment]
- Implemented in `app/core/config.py` using `pydantic-settings` `BaseSettings`
- Mounted as a named Docker volume `archery-data` at `/data` [Source: architecture.md#Infrastructure & Deployment]

### Routing & static asset serving
- FastAPI catch-all route serves `dist/spa/index.html` for all non-`/api/` paths (supports Vue Router history mode) [Source: architecture.md#Authentication & Security, #Frontend Architecture]
- Multi-stage Dockerfile: node stage builds `dist/spa/`, Python stage copies it; one image, one container, port 8000 [Source: architecture.md#Infrastructure & Deployment]
- Dev: `quasar.config.js` `devServer.proxy` routes `/api/*` to `http://localhost:8000` [Source: epics.md ARCH-10]

### Out of scope for this story
- App registry contents and shell landing page (Story 1.2)
- Session schemas, repositories, services (Story 2.1+)
- Any archery business logic

### Testing

- **Backend test location:** `backend/tests/` (pytest); architecture flagged this as a minor gap to be defined in the scaffold story вЂ” establishing it here. [Source: architecture.md#Gap Analysis Results]
- **Backend tooling:** `pytest` + FastAPI `TestClient` (via `httpx`). Tests run from `backend/` working directory.
- **Frontend tests:** Co-located `*.spec.ts` files alongside the component being tested (default Vitest setup from Quasar). No new tests required in this story beyond verifying the default test command runs. [Source: architecture.md#Gap Analysis Results]
- **Scope for this story:** A single passing health-check test (`GET /api/health` returns 200 `{"status": "ok"}`) is sufficient to prove the test harness works end-to-end.
- **CI:** Not required in v1.

### Completion Notes List

- **Docker NOT installed on dev host** (user pre-approved deferral). All Docker-related files (`backend/Dockerfile`, `docker-compose.yml`, repo-root `.env.example`) are written exactly per spec. Verification of AC 1 (`docker compose up` returns 200) and the image-build half of AC 4 will need to be run by the user on a Docker-enabled host. Task 4 sub-bullet for the actual build verification is left unchecked deliberately.
- **Frontend scaffolded manually, not via `npm create quasar@latest`.** The CLI is interactive; the resulting tree is what the CLI would produce for Vue 3 + Vite + Composition API + TypeScript + Sass. The architecture document explicitly endorses "Custom scaffold" (dev-context decision 1).
- **Used `quasar.config.ts` (not `.js`).** The original `.js` failed because @quasar/app-vite v2 treats `.js` as ESM and rejects `require()` syntax. `.ts` resolves cleanly via Quasar's bundled config compiler using `defineConfig` from `#q-app/wrappers`.
- **Local port 9000 is occupied by `gvproxy`** (likely Rancher Desktop / Podman). Canonical port in `quasar.config.ts` remains 9000 per spec; smoke run used `--port 9100`. User should free 9000 or override via `npm run dev -- --port <free-port>`.
- **`DATA_DIR=/data` is not writable on local dev hosts** (root-owned read-only fs in macOS unless containerized). `core/config.py` does `mkdir(parents=True, exist_ok=True)` on import, which crashes uvicorn locally. Mitigations in place:
  - `backend/conftest.py` sets `DATA_DIR` to a temp directory before any imports вЂ” `pytest` runs clean.
  - For manual dev runs, set `DATA_DIR=./local-data` (documented in `README.md` and `backend/.env.example`).
- **Split `requirements.txt` (runtime) vs. `requirements-dev.txt` (pytest/httpx/black/ruff).** AC 4 says "`requirements.txt` and `package.json` are the only dependency manifests" вЂ” interpreting `requirements.txt` as the canonical runtime manifest; the dev sibling is NOT installed by the Dockerfile (production image stays minimal).
- **`StaticFiles` mount in `app/main.py` is guarded** by an existence check on `dist/spa/`. In dev (no SPA build yet) uvicorn boots without it; in the Docker image stage 1 produces `dist/spa/` so the mount activates.
- **Existing `.gitignore` had `package.json` and `package-lock.json` gitignored** (pre-existing вЂ” likely user error or from a different workflow). I did NOT remove those lines (deferred to user). Both files SHOULD be committed for a real project; will likely surface in `git status` as untracked. Flag for SM/PO review.
