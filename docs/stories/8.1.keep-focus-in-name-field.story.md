# Story 8.1: Keep Focus in Name Field After Add

## Status

Draft

## Story

**As an** operator,
**I want** the cursor to stay in the archer-name field after I tap Add,
**so that** I can enter several archers in a row without re-tapping the field.

## Acceptance Criteria

1. **Given** the session setup screen, **When** the operator adds an archer (via the Add button or Enter), **Then** the name input is cleared and keeps/regains keyboard focus.
2. **Given** focus is restored, **When** the operator types immediately, **Then** the next name lands in the input with no extra tap.
3. **Given** an add is rejected (empty or duplicate), **When** the error shows, **Then** focus also remains in the input for correction.

## Tasks / Subtasks

- [ ] **Task 1 — Template ref on the input** (AC: 1, 2, 3)
  - [ ] In `SessionSetupPage.vue`, add a template ref to the name `q-input`, e.g. `const nameInput = ref()` and `ref="nameInput"` on the `<q-input>`.

- [ ] **Task 2 — Refocus after add** (AC: 1, 3)
  - [ ] In `addArcher()`, after both the success path (push + clear) and the rejection paths (empty/duplicate error), call `nameInput.value?.focus()` inside `nextTick()` so focus survives the re-render.
  - [ ] Import `nextTick` from `vue`.

- [ ] **Task 3 — Tests** (AC: 1, 2, 3)
  - [ ] `SessionSetupPage.spec.ts`: after a successful add, the input is empty and `focus()` was invoked; after a duplicate/empty add, `focus()` was invoked. (Spy on the input component's `focus`, or assert document.activeElement in a mounted-to-DOM test.)

## Dev Notes

### Previous Story Insights
- Story 2.2 built `SessionSetupPage.vue` with `addArcher()` (validates, pushes to `store.draftRoster`, clears `store.draftName`) bound to the Add button and `@keyup.enter`. Focus is currently lost after Add because the value clears and nothing re-focuses.
- Quasar's `q-input` exposes a `focus()` method on its component instance — call it via the template ref.

### Source references
- `frontend/src/apps/archery/pages/SessionSetupPage.vue` — `addArcher()`, the name `q-input` (`v-model="store.draftName"`), `inputError`.
- Note: Story 6.2 adds a *separate* "Session name" input to this page — do not confuse it with the archer-name input. This story targets the archer-name input only.

### Constraints / anti-patterns
- ❌ Don't auto-focus on initial mount in a way that pops the mobile keyboard unexpectedly before the operator engages — only refocus as a response to an Add action.
- ❌ Keep the existing validation behaviour intact (empty/duplicate messages).

### Testing
- **Frontend:** Vitest + `@vue/test-utils`. Use `attachTo: document.body` if asserting real focus, or spy on the exposed `focus` method via a stub/ref. Wrap focus assertions after `await nextTick()`.
